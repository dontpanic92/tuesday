17.1Quantization：int8 / int4 / GGUF

当前 xinfer 使用 f16 weights 与 f32 activations。下一步最自然的优化是量化：用更少 bit 表示权重， 进一步减少显存和带宽。

格式字节/权重优点代价
f162简单，硬件支持好仍较大
int81显存/带宽再减半需要 scale / zero-point
int40.5极省显存，适合大模型解包复杂，精度风险更高

weight-only quantization 只量化权重，activation 仍用 f16/f32。典型反量化公式：

\[ w \approx s \cdot (q - z) \]

其中 \(q\) 是整数权重，\(s\) 是 scale，\(z\) 是 zero-point。很多 LLM 推理格式（例如 GGUF 的若干量化类型）会按 block 存 scale，使不同权重块有不同缩放因子。

17.2Tiled GEMM 与 tensor-core / WMMA 路径

当前 coalesced GEMV 已经比 naive kernel 快很多，但仍然不是“矩阵乘法的终点”。进一步优化会使用 tile： 一个 threadgroup 处理 \(M\times N\) 的小块，把 A 和 W 的 tile 缓存在 groupshared，然后重复利用。

\[ C_{tile} = A_{tile} W_{tile}^{\top} \]

对大 \(M\) 的 prefill，tiled GEMM 可以显著提高 arithmetic intensity。对 decode \(M=1\)，重点则是缓存 activation vector、 同时计算多个输出列，减少重复读取 \(x\)。

Tensor-core / WMMA 类路径需要更现代的 shader 能力（例如 DXC / SM6.x 的 wave/tensor 操作），这会带来更高性能，也会增加平台和编译复杂度。

17.3Fused epilogues：把小算子合进大算子

当前 pipeline 里很多小算子会产生中间 buffer：linear → bias_add → RoPE，或 gate/up → SwiGLU → down。 如果把 bias、activation、residual 等操作融合到 matmul 的 epilogue 中，可以减少：

  • dispatch 次数；
  • 中间 buffer 写入/读取；
  • UAV barrier；
  • 显存带宽。

例如 Q projection 可以从：

\[ Q = XW_Q^\top,\qquad Q'=Q+b_Q \]

融合成：

\[ Q'_j = \sum_p X_p W_{Q,j,p}+b_{Q,j} \]

SwiGLU 也可以作为 gate/up matmul 后的融合 epilogue，但因为它需要两个 matmul 的结果，设计会更复杂。

17.4GPU sampling 与 speculative decoding

当前 xinfer 对 greedy 使用 GPU argmax，但 temperature/top-k/top-p 仍然需要读回完整 logits。 下一步可以把 top-k / top-p 也搬到 GPU：

  • GPU partial top-k：在 GPU 上保留少量候选 token；
  • 只读回 top-k token ids 和 logits，而不是完整 vocab；
  • CPU 或 GPU 完成最终随机采样。

Speculative decoding 则是另一类优化：用一个小模型先草拟多个 token，再用大模型验证。理想情况下，每次大模型 forward 可以接受多个候选 token， 从而减少自回归循环次数。

17.5多序列 batching 与 paged KV cache

单用户本地 demo 只需要一个 sequence；服务端推理通常要同时服务多个请求。Batching 把多个 sequence 合在一起跑， 提高 GEMM 利用率，但也带来复杂度：

  • 不同请求 prompt 长度不同；
  • 不同请求生成长度不同；
  • KV cache 需要动态分配和回收；
  • attention mask 变得更复杂。

Paged KV cache 把 KV cache 切成固定大小的 page，像虚拟内存一样管理，避免为每个请求预留最大长度连续 buffer。

\[ \text{logical token position} \rightarrow (\text{page id},\text{offset within page}) \]
从教学引擎到生产引擎 xinfer 当前是单序列、可读性优先的教学引擎。要走向生产系统，batching、paged KV、调度器和量化会变得非常重要。

思考与练习

基础weight-only quantization 与 activation quantization 有什么区别？

weight-only 只把权重量化到低位（如 int4/int8），激活仍保持 f16/f32，计算时把权重解量化回浮点再算——主要省显存与权重读取带宽，对 decode 这种带宽受限场景特别有效，精度损失小、易实现。activation quantization 还把激活也量化，能用整数矩阵乘进一步加速计算，但激活分布动态范围大、更难量化，掉点风险高，常需 per-tensor/per-token scale 与校准。LLM 推理常先采用 weight-only。

基础为什么 int4 权重能显著减少带宽？

int4 每个权重 0.5 字节，是 f16 的 1/4、f32 的 1/8。decode 受限于把权重从显存读进计算单元的带宽，读取字节数直接随位宽缩小，所以 int4 理论上比 f16 再快约 4×（扣除解量化开销）。同时显存占用也降到 1/8，可容纳更大模型。代价是需要 group-wise scale/zero-point 和解量化 kernel，并接受一定精度损失。

进阶举例说明 fused epilogue 如何减少一次中间 buffer 写读。

例：MLP 的 SwiGLU。未融合时 gate 投影输出写回显存、up 投影输出写回显存，再启动一个 elementwise kernel 读回两者算 silu(gate)*up 又写回。融合后，在 matmul kernel 的尾段（epilogue）算出 gate/up 结果时，直接在寄存器里做 silu(gate)*up 再写一次结果，省掉中间两次写 + 一次读 + 一个 dispatch。类似地可把 bias、residual add 融进 matmul epilogue。每省一次往返就减少一份带宽与一次提交，对带宽受限的 decode 累积起来可观。

进阶GPU top-k 相比 GPU argmax 多了哪些难点？

argmax 只需一个 max 归约（结合律好、单值、一轮 WaveActiveMax/LDS 即可）。top-k 要维护“前 k 大”的有序集合：①每个线程/组要保留 k 个候选而非 1 个（堆或排序网络，寄存器/LDS 压力大）；②跨组合并多个局部 top-k 成全局 top-k 需要第二阶段归约，且合并不是简单求 max；③要同时跟踪 value 和 原始 index；④处理并列、k 与 group 大小不整除等边界。复杂度和实现成本都明显高于 argmax。

挑战设计 paged KV cache 的数据结构，并说明 attention kernel 如何根据 page table 读取 K/V。

数据结构：把 KV cache 切成固定大小的 page（如每 page 存 16 个 token 的 K/V）。维护一个 page pool（一堆等大的 GPU block）和每个序列的 page_table：一个数组，第 \(b\) 项给出该序列第 \(b\) 个逻辑块对应的物理 page id。token 位置 \(p\) → 逻辑块 \(b=\lfloor p/\text{page\_size}\rfloor\)，块内偏移 \(o=p\bmod\text{page\_size}\)。

kernel 读取：attention 遍历 key 位置 \(p\) 时，不再用连续地址，而是：查 page_table[b] 得到物理 page，基址 = page_base + \(o\times\)（每 token 的 K/V 字节）。即一次额外的间接寻址。好处：①显存按需分配、几乎零碎片（vLLM 思路），能服务更多并发序列；②序列间可共享 page（如相同前缀）。代价是每次 K/V 访问多一层 page_table 查找，且 kernel 要传入 page_table 与 page_size。
