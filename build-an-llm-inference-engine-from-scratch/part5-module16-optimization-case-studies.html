16.1总览：从 3.6 到 30.9 tok/s

本项目的优化不是一次“神奇加速”，而是分层排除瓶颈。每一层都解决一个明确问题，并用 parity tests 保证输出没有变。

16.2Tier 1：把 dispatch 批到一个 command list

Baseline 的 decode 在每个小算子后都 submit_and_wait。一个 token 大约有：

\[ 22\;\text{ops/layer}\times24\;\text{layers}\approx528 \]

次 GPU submission。每次 submission 都有 CPU/GPU 同步成本。Tier 1 的做法是：一个 layer 内连续记录所有 dispatch， 用 UAV barrier 保证依赖顺序，最后只 submit 一次。

\[ 528\;\text{submits/token}\quad\longrightarrow\quad 24\;\text{submits/token} \]

这不改变任何数学，只改变命令提交方式，因此输出 bit-identical。收益：3.6 → 8.5 tok/s。

16.3Tier 2：f16 weights，带宽与显存减半

Decode 的 matmul/GEMV 多数是 memory-bound。把权重从 f32 改成 f16 后，每个权重从 4 bytes 变成 2 bytes：

\[ \operatorname{bytes}_{f16}(W) = \frac{1}{2}\operatorname{bytes}_{f32}(W) \]

xinfer 使用 ByteAddressBuffer + f16tof32 解包权重，activation 仍保持 f32。 权重显存从 1885 MiB 降到 942 MiB，decode 从 8.5 → 10.3 tok/s。

数值注意 f16 权重会带来舍入误差，因此 parity test 需要合理容差，并检查 argmax 是否稳定。

16.4Tier 3：Coalesced GEMV + groupshared reduction

Naive GEMV 是“一个线程算一个输出元素”。这样相邻线程读取的权重地址相隔 \(K\)，不利于 coalescing。 Tier 3 改成“一个 threadgroup 算一个输出元素”：组内线程沿 \(K\) 方向读取相邻权重，然后在 groupshared 中归约。

\[ y_j=\sum_{p=0}^{K-1}x_pW_{j,p} \]

每个线程负责 \(p=t,t+TG,t+2TG,\ldots\) 的部分和。这样同一 wave 中的线程更可能读取连续权重。 LM head 输出维度超过 65535，因此还需要 2D dispatch grid。

16.5Tier 4：Flash-style single-pass attention

传统稳定 softmax 会先扫一遍 key 找最大值，再扫一遍计算权重与加权 V。Tier 4 使用 online softmax： 一次遍历维护 running max、denominator 和 accumulator。

\[ m'=\max(m,s),\qquad d'=d\cdot e^{m-m'}+e^{s-m'} \]

accumulator 同样乘以 \(e^{m-m'}\) 做 rescale，然后加上当前 \(e^{s-m'}v\)。 对短上下文收益不大，因为 attention 不是瓶颈；但长上下文时它减少 K 读取，价值会更明显。

16.6Tier 5：Cooperative RMSNorm

原始 RMSNorm 如果一个线程处理一整行，decode 时每层有两个串行 896 元素 reduction。Tier 5 改成一个 threadgroup 处理一行： 线程分摊平方和，再用 groupshared reduction 合并。

\[ \sum_{j=1}^{H}x_j^2 = \sum_{\text{threads }t}\left(\sum_{j\in \mathcal{J}_t}x_j^2\right) \]

这类优化收益不如 Tier 1 / Tier 6 显眼，但它消除了明显的串行热点，也为后续 fused norm kernel 打基础。

16.7Tier 6：GPU-side argmax

这是最出乎意料的瓶颈之一。Qwen2.5 的词表大小是 151936。若 logits 是 f32，每步读回：

\[ 151936\times4\approx608\text{ KB} \]

Greedy decoding 其实只需要最大 logit 的 index。Tier 6 增加 GPU argmax kernel，把最后一行 logits 在 GPU 上归约成一个 u32 token id，只读回 4 bytes。

\[ 608\text{ KB/token}\quad\longrightarrow\quad4\text{ bytes/token} \]

结果 decode 从 12.3 → 30.9 tok/s。这个优化不减少模型计算量，却显著减少 CPU/GPU 传输和 CPU 扫描成本。

思考与练习

基础哪一个 tier 主要减少 GPU submission？哪一个主要减少 readback？

T1（把整层前向录进一个命令列表、每层提交一次）主要减少 GPU submission——从约 530 submits/token 降到约 24。T6（GPU 端 greedy argmax）主要减少 readback——从 608 KB/token 降到 4 字节。

基础为什么 f16 权重能同时降低显存和带宽？

f16 每个权重 2 字节，是 f32 的一半。存储减半 → 显存从 1885 降到 942 MiB；而 decode 是带宽受限的，每次都要把整组权重从显存读进计算单元，读取字节减半 → 有效带宽翻倍，matmul 更快。一举两得，代价是少量精度损失（用 fp32 累加 + 放宽容差，argmax 仍稳定）。

进阶为什么 attention 优化在短上下文下收益不明显？

因为 attention 的成本随上下文长度 \(T\) 增长，而短上下文时 K/V 很少，attention 在总耗时中占比很小——瓶颈是各 matmul（QKV/MLP/LM head）。flash-style 单遍 attention 省的是对 K/V 的重复扫描带宽，\(T\) 小时这点带宽微不足道。只有上下文变长（成百上千 token）后，attention 占比上升，单遍 + online softmax 的收益才显著。所以 xinfer 在短 ctx 下测得 T4≈中性。

进阶如果使用 temperature sampling，GPU argmax fast path 还能用吗？为什么？

不能直接用。temperature/top-k/top-p 需要完整（或至少 top-k 的）概率分布来做带随机性的采样，而 GPU argmax 只返回单个最大下标，丢掉了分布信息。所以带温度采样时要么读回完整 logits 在 CPU 采样，要么实现 GPU 端 top-k 只回传少量候选。greedy（等价 temperature=0）才只需 argmax，能走 4 字节 fast path。

挑战设计一个 GPU top-k 方案，估计它能减少多少 readback。

方案：在 GPU 上对 151936 个 logits 做并行 top-k。①每个 threadgroup 处理一段 logits，用局部堆/排序网络维护本段 top-k；②跨组归约合并各段的 top-k 得到全局 top-k（k 很小，可用一轮 LDS 归约或第二个 kernel）；③只把 k 对 (logit, index) 拷回 CPU，在 CPU 做温度缩放 + 归一化 + 采样。

readback 估计：原来回传全部 logits = \(151936\times4\approx608\) KB。top-k 只回传 \(k\) 个 (f32 logit + u32 index) = \(k\times8\) 字节。取 \(k=50\)：约 400 字节，相比 608 KB 减少约 1500×（与 argmax 的 4 字节同量级）。代价是 top-k kernel 本身的复杂度与一点额外计算，但对带宽/PCIe 受限的 decode 是大赢。
