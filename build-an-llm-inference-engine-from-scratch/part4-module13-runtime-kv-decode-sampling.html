Module 13：Phase 4 — Runtime：KV Cache、Decode 与 Sampling

把“完整 forward”变成真正会说话的 runtime：per-layer KV cache、prefill/decode 循环、采样器、tokenizer 与第一次真实 Qwen2.5 demo。

学习目标

  • 实现 per-layer KV cache，并理解每个 layer 都有自己的 K/V 缓存。
  • 理解 CopyBufferRegion 如何把新 token 的 K/V 追加到 cache。
  • 把 attention kernel 泛化到 seq_q != seq_k，支持 prefill 与单步 decode。
  • 实现 prefill → autoregressive decode loop。
  • 实现 greedy、temperature、top-k、top-p 采样和可复现 RNG。
  • 集成 HuggingFace tokenizer、Qwen chat template 与 EOS token。

13.1Per-layer KV cache：每层一份 K/V

Phase 3 的 forward 已经能对一整个 token 序列输出 logits，但如果每生成一个 token 都重新 forward 全部历史， 计算会大量重复。Phase 4 的核心是引入 KV cache：每一层保存过去 token 的 key 和 value。

对第 \(\ell\) 层，cache 结构可以写成：

\[ K^{(\ell)}_{\text{cache}}\in\mathbb{R}^{T\times n_{\text{kv}}\times d_{\text{head}}}, \qquad V^{(\ell)}_{\text{cache}}\in\mathbb{R}^{T\times n_{\text{kv}}\times d_{\text{head}}} \]

其中 \(T\) 是最大 cache capacity。新 token 进入第 \(\ell\) 层时，模型只计算这个 token 的 \(K_{\text{new}},V_{\text{new}}\)， 然后把它们写到第 pos_base 个槽位：

\[ K^{(\ell)}_{\text{cache}}[\text{pos}] \leftarrow K^{(\ell)}_{\text{new}}, \qquad V^{(\ell)}_{\text{cache}}[\text{pos}] \leftarrow V^{(\ell)}_{\text{new}} \]

在 xinfer 中，这个追加用 D3D12 CopyBufferRegion 完成，因为新 K/V 已经是连续的 GPU buffer， cache 中对应位置也是连续的 byte range。

13.2泛化 attention：query 长度与 key 长度可以不同

没有 KV cache 时，attention 常常假设 \(seq_q=seq_k=S\)。但有了 cache 后，decode 一步通常是：

\[ seq_q = 1,\qquad seq_k = \text{past length}+1 \]

因此 attention kernel 必须接受两个长度：seq_q 与 seq_k。对 query row \(t_q\)，它的绝对位置是 \(q\_\text{pos\_base}+t_q\)，可见 key 数为：

\[ \text{key\_count}=\min(q\_\text{pos\_base}+t_q+1,\;seq_k) \]

这个公式同时支持 prefill 和 decode：prefill 时 \(q\_\text{pos\_base}=0\)，decode 时它等于 cache 中已有 token 数。

13.3Prefill → decode loop

Runtime 的生成流程可以写成以下伪代码： 

fn generate(prompt_tokens, max_new_tokens):
    cache = model.make_cache(prompt_len + max_new_tokens)

    // 1. prefill：一次处理整个 prompt
    logits_or_argmax = model.decode(cache, prompt_tokens, pos_base=0)
    next = sample(last_logits)

    // 2. decode：一次追加一个 token
    output = []
    for step in 0..max_new_tokens:
        if next == eos: break
        output.push(next)
        logits_or_argmax = model.decode(cache, [next], pos_base=cache.len)
        next = sample(last_logits)

    return tokenizer.decode(output)

注意 model.decode 这个名字在工程上表示“把 tokens 接到 KV cache 后执行一次 forward”。 它既可以用于 prefill（输入很多 token），也可以用于单步 decode（输入一个 token）。

13.4Sampling：Rust 中的可复现采样器

采样器把 logits 转成下一个 token。Phase 4 实现了：

  • Greedy：取最大 logit；
  • Temperature：缩放 logits；
  • Top-k：只保留最高的 \(k\) 个候选；
  • Top-p：只保留累积概率达到 \(p\) 的最小候选集合；
  • 可复现 RNG：使用简单 xorshift，使实验结果可重复。

temperature 公式：

\[ p_i = \operatorname{softmax}\left(\frac{z_i}{\tau}\right) \]

top-p 集合：

\[ \mathcal{S}_p = \min\left\{S:\sum_{i\in S}p_i\ge p\right\} \]
工程优化 默认 greedy 路径后来被优化成 GPU argmax：runtime 不再读回完整 logits，而只读回一个 u32 token id。 非 greedy 采样仍然读回 logits，在未来可以继续做 GPU top-k / top-p。

13.5Tokenizer 与 Qwen chat template

Runtime 需要在文本和 token id 之间转换。Phase 4 的 xinfer-tokenizer 封装 HuggingFace tokenizers： 

let tokenizer = Tokenizer::from_dir(model_dir)?;
let ids = tokenizer.encode(prompt_text, false)?;
let text = tokenizer.decode(&generated_ids, true)?;

对 instruct 模型，还需要 Qwen chat template： 

<|im_start|>system
You are a helpful assistant.<|im_end|>
<|im_start|>user
... user prompt ...<|im_end|>
<|im_start|>assistant

EOS token id 通常来自 generation_config.json 或 config.json。如果生成到 EOS，runtime 停止循环。

13.6第一次真实端到端 demo

Phase 4 的里程碑是：真实 Qwen2.5 checkpoint、真实 tokenizer、GPU model、KV cache、采样器全部连起来， 命令行能输出文本。 

xinfer generate --model models\Qwen2.5-0.5B-Instruct \
  --prompt "What is the capital of France?" --max-tokens 16

The capital of France is Paris.

从系统角度看，这条命令经过了完整路径：

Lab 13生成文本，并验证 incremental decode == full forward

本实验有两个验收目标。第一，真实模型能生成文本。第二，KV cache 的 incremental decode 必须与 full forward 的最后一行 logits 匹配。 

# 真实模型生成
cargo run --release -p xinfer-cli -- generate \
  --model models\Qwen2.5-0.5B-Instruct \
  --prompt "What is the capital of France?" \
  --max-tokens 16

# KV cache correctness
cargo test -p xinfer-model --test forward_parity -- --nocapture

正确性测试的数学意义是：

\[ \operatorname{logits}_{\text{full}}[S-1] \approx \operatorname{logits}_{\text{prefill+decode}}[0] \]

也就是说，先一次性 forward 全部 tokens，和先 prefill 前 \(S-1\) 个 token 再 decode 最后一个 token，最后位置的 logits 应该一致。

思考与练习

基础为什么每一层都需要自己的 KV cache？

因为每层有各自独立的 K/V 投影权重，产生不同的 key/value 表示。第 \(\ell\) 层的 attention 只能用第 \(\ell\) 层历史 token 的 K/V，不能跨层共用。所以要为 24 层各保存一份 KV cache，decode 时各层把新 token 的 K/V 追加到对应层的缓存里。

基础prefill 与 decode 调用的是同一个 model.decode，为什么输入 shape 不同？

prefill 一次喂入整个 prompt，seq_q = prompt 长度（如 128），同时把这 128 个 K/V 写入空缓存（seq_k 也是 128）。decode 每步只喂 1 个新 token，seq_q=1，而 seq_k = 已有历史长度+1。同一函数通过 seq_q≠seq_k 与 q_pos_base 两个参数统一处理两种情形，attention kernel 据此设置 causal 上限。

进阶推导 key_count = min(q_pos_base + tq + 1, seq_k) 的含义。

query 在序列中的绝对位置是 \(p=q\_pos\_base+t_q\)（tq 是本批内的相对索引）。causal 规则下它能看见位置 \(0..p\)，即 \(p+1\) 个 key，故可见数为 \(q\_pos\_base+t_q+1\)。再对 seq_k（缓存中实际存在的 key 数）取 min，防止越界。decode 时 \(t_q=0\)，\(q\_pos\_base\) = 历史长度，于是 key_count = 历史长度+1 = 全部缓存，符合“新 token 能看到所有过去 + 自己”。

进阶解释 greedy fast path 为什么可以只读回一个 u32。

greedy 解码只需要 argmax(logits) 这一个 token id，不需要整条概率分布。于是在 GPU 上做归约求出最大 logit 的下标，只把这个 4 字节的 u32 拷回 CPU，而不是把 151936 个 f32（约 608 KB）全读回再在 CPU 上找 max。这把每 token 的 readback 从 608 KB 降到 4 字节，是 T6 优化让 decode 提到 30.9 tok/s 的关键一环。温度/top-k/top-p 采样仍需更多信息，故保留 CPU 路径。

挑战实现一个测试：随机 prompt 下 full forward 最后一行 logits 与 incremental decode logits 的 max abs diff 小于阈值。

步骤：① 随机生成一段 token 序列作为 prompt；② 路径 A：对完整序列做一次 full forward，取最后一个位置的 logits；③ 路径 B：先 prefill 前 \(n-1\) 个 token 建立 KV cache，再对第 \(n\) 个 token 调 decode 拿 logits；④ 断言 max(abs(A - B)) < eps。

理论上 incremental decode 与 full forward 在数学上等价，f32 路径 diff 应为 0（xinfer 的 kv_cache_decode_matches_full 实测 diff=0）；若用 f16 kernel 则放宽到 ~1e-2。这个测试守护 KV cache 追加、位置编码偏移与 causal mask 的正确性——它们是 decode 最易出错的环节。
