<h2><span className="num">8.1</span>DirectML 提供什么，我们自己写什么？</h2>

DirectML 是 Windows / Xbox 生态中的机器学习算子 API。它提供诸如 GEMM、卷积、softmax、elementwise 等 operator。
    但 DirectML 不是一个完整 LLM runtime：它不会帮你处理 tokenizer、Qwen2 tensor naming、KV cache、采样、layer streaming、
    或者自定义 fused kernel。

<table className="tbl">
    <thead><tr><th>能力</th><th>DirectML operator</th><th>自写 HLSL kernel</th></tr></thead>
    <tbody>
      <tr><td>通用矩阵乘 / softmax</td><td>有现成 operator</td><td>可针对 shape 定制</td></tr>
      <tr><td>RMSNorm / RoPE / SwiGLU</td><td>通常需要组合多个 operator</td><td>一个 kernel 直接实现</td></tr>
      <tr><td>KV cache append</td><td>不是典型 ML operator</td><td>CopyBufferRegion 或专用 kernel</td></tr>
      <tr><td>GPU argmax</td><td>可用 reduce 类 operator 或自写</td><td>简单、可控、少 readback</td></tr>
      <tr><td>异常 shape / driver quirk</td><td>可能遇到黑盒问题</td><td>可绕开并调试</td></tr>
    </tbody>
  </table>

xinfer 的策略是：使用 D3D12 作为资源和执行基础；DirectML 用于研究 operator 生命周期和部分实验；
    关键 transformer 路径（linear、RMSNorm、RoPE、attention、SwiGLU、argmax）则使用自写 HLSL kernel。

<h2><span className="num">8.2</span>DirectML operator 生命周期</h2>

DirectML operator 不是创建后立刻 dispatch。它有一个比较严格的生命周期：

1. **Create：**用 operator descriptor 描述输入/输出 tensor shape、dtype、参数。
2. **Compile：**把 operator 编译成可执行的 dispatchable。
3. **Initialize：**有些 operator 需要 persistent / temporary resource 初始化。
4. **Bind：**创建 binding table，把 input/output/temp/persistent buffer 绑定进去。
5. **Dispatch：**通过 DML command recorder 把 operator 记录到 D3D12 command list。

这套流程适合“成熟的通用 operator”。但如果你需要 fuse 多个小算子，或处理特殊 layout，自写 HLSL 往往更直接。

<h2><span className="num">8.3</span>什么时候用 DML operator，什么时候写 custom kernel？</h2>

一个实用判断标准是：如果 operator 足够通用、shape 大、库实现可靠，优先考虑库；如果需要特殊布局、融合、轻量控制或绕开 driver 问题，
    就写自定义 kernel。

<table className="tbl">
    <thead><tr><th>情况</th><th>倾向</th><th>原因</th></tr></thead>
    <tbody>
      <tr><td>大 GEMM，库实现稳定</td><td>DML / vendor lib</td><td>库可能有高度优化的 tiling</td></tr>
      <tr><td>RMSNorm / RoPE</td><td>HLSL</td><td>组合 DML operator 会产生多次 dispatch / 中间 buffer</td></tr>
      <tr><td>KV cache append</td><td>D3D12 copy 或 HLSL</td><td>更像内存操作，不是典型 ML op</td></tr>
      <tr><td>f16 packed weights / custom layout</td><td>HLSL</td><td>需要手动解包和 coalesced 访问</td></tr>
      <tr><td>driver / device removed</td><td>HLSL</td><td>黑盒 operator 不可控，自写路径可调试</td></tr>
    </tbody>
  </table>

<h2><span className="num">8.4</span>案例：DirectML GEMM 与 RDNA4 device-removal</h2>

本项目最重要的 backend 决策来自一个真实问题：在 AMD RX 9070 XT（RDNA4）上，DirectML 的 GEMM operator
    对某些 shape 会触发 device removed（HRESULT `0x887A0005`）。例如 $M=5,K=64,N=32$ 的 GEMM 会失败。

<Callout kind="warn" tag="调试观察">

D3D12 debug layer 输出过 “CreateMetaCommand parameters are not supported” 警告，说明 DirectML 的 metacommand 路径不可用，
    回退路径在该驱动/shape 上有问题。自写 HLSL matmul 则正常工作。

</Callout>

这迫使我们把 matmul 改成 custom HLSL kernel。后来我们进一步优化它：f16 权重、coalesced GEMV、groupshared reduction、
    2D dispatch grid、GPU argmax。这个案例有两个教学价值：

1. 库 operator 不是“数学正确就一定可靠”；实际系统会受 driver、硬件、shape、dtype 影响。
2. 拥有自写 kernel 能力后，你可以绕开黑盒故障，并做针对模型 shape 的优化。

因此 xinfer 的 Rust core 支持两条 device path：

    第二步：尝试一个已知可能触发 device-removal 的 shape，观察错误，并用自写 HLSL matmul 路径绕开。

```bash
# 运行 DML / HLSL 相关测试
cargo test -p xinfer-dml --test gemm_smoke

```

实验报告应回答：

1. DML GEMM operator 需要哪些 descriptor 和 binding？
2. CPU reference 如何计算同一个结果？误差阈值应该怎么设？
3. 如果 device removed，如何用 D3D12 debug layer 和 `GetDeviceRemovedReason` 缩小问题范围？
4. 为什么自写 HLSL kernel 可以绕开该问题？它牺牲了什么，又获得了什么？

<QAGroup>

### 思考与练习

<QA question={<><span className="diff easy">基础</span>列出 DirectML operator 的五个生命周期阶段。</>}>

① create（按 desc 创建 operator）；② compile（编译成可执行的 compiled operator）；③ initialize（运行 initializer，分配/填充 persistent 资源）；④ bind（绑定输入/输出/temporary/persistent buffer 到 binding table）；⑤ execute（录制到命令列表并提交执行）。xinfer 的 `DmlOperator` 封装正是这五步。

</QA>

<QA question={<><span className="diff easy">基础</span>解释 SRV/UAV/temp/persistent buffer 在 DML binding 中的角色。</>}>

SRV（只读输入，如权重/激活）；UAV（可写输出）；temporary buffer 是算子执行期间的临时工作区（execute 后内容无意义，可复用）；persistent buffer 在 initialize 时填充、execute 时反复读取（如某些算子预处理后的常量），生命周期跨多次 execute，需一直保留。绑定时要按算子要求的大小分别提供。

</QA>

<QA question={<><span className="diff mid">进阶</span>为什么 RMSNorm 更适合写成 custom kernel，而不是组合多个 DML operator？</>}>

用 DML 拼 RMSNorm 需要 square→reduce-mean→rsqrt→broadcast-mul→scale 等多个算子，每步都把整行激活写回显存再读出，产生大量中间往返与多次 dispatch。写成一个 custom HLSL kernel 可在 groupshared 内一次完成归约并就地缩放，省去中间 buffer 与多次提交，延迟更低；而且能和后续算子融合。对 decode（seq=1）这种延迟敏感场景收益尤其明显。

</QA>

<QA question={<><span className="diff mid">进阶</span>解释 PC 和 Xbox device creation 的关键差异。</>}>

PC 走 DXGI：枚举 adapter，用 `D3D12CreateDevice` 创建设备。Xbox 主机没有 DXGI，用专用入口 `D3D12XboxCreateDevice`（带主机特定参数），并使用固定的主机 GPU。xinfer 的应对是把设备创建抽到外部，提供 `DmlDevice::from_existing/from_raw_pointers` 让宿主创建好 `ID3D12Device`+queue 后注入引擎，从而同一套 Rust 核心同时支持 PC 与 Xbox。

</QA>

<QA question={<><span className="diff hard">挑战</span>为一个新算子制定决策：用 DML 还是 HLSL？写出你考虑的 shape、dtype、融合、debug 风险。</>}>

<span className="label">用 DML 当：</span>算子是标准操作（GEMM、softmax、卷积），shape 常规且 DML 有高度优化实现，dtype 受支持，无需特殊融合。省去手写与调试。

<span className="label">用 HLSL 当：</span>① 算子 DML 没有或语义不符（RMSNorm、RoPE、SwiGLU、KV append）；② 需要算子融合减少往返（如 silu(gate)*up、bias 融入 matmul 尾段）；③ 某些 shape/dtype 在该硬件上触发 DML bug——本项目就遇到 DML GEMM 在 RDNA4 特定 shape 下移除设备（0x887A0005），只能改用 custom matmul。

<span className="label">风险权衡：</span>shape（异常/小 batch 是否触发 DML 慢路径或 bug）、dtype（f16 解包是否需要自己处理）、融合（能否显著减少 dispatch）、debug（custom kernel 需自己写 CPU 参考 + parity 测试，成本更高但可控）。结论：能用 DML 就用，遇到缺失/融合/硬件 bug 才落到 HLSL，并配 parity 测试守护正确性。

</QA>

</QAGroup>
