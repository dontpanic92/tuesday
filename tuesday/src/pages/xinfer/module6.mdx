<h2><span className="num">6.1</span>HLSL compute shader 的最小形态</h2>

Compute shader 是一个 GPU 程序。它没有传统意义上的 `main(argc, argv)`；
    它的“输入”来自绑定的 GPU buffer 和 constant buffer，它的执行规模来自 CPU 侧的 dispatch。

```hlsl
// shaders/affine.hlsl：out[i] = in[i] * 2 + 1
StructuredBuffer<float> In  : register(t0);
RWStructuredBuffer<float> Out : register(u0);

cbuffer Params : register(b0)
{
    uint count;
};

[numthreads(64, 1, 1)]
void main(uint3 tid : SV_DispatchThreadID)
{
    uint i = tid.x;
    if (i < count)
        Out[i] = In[i] * 2.0f + 1.0f;
}
```

`[numthreads(64,1,1)]` 表示每个 threadgroup 有 64 个线程。若 CPU 侧 dispatch $G$ 个 group，
    总线程数最多是 $64G$。每个线程用 `SV_DispatchThreadID` 得到自己的全局编号。

$$
\text{global\_thread\_id}
  = \text{group\_id}\cdot\text{numthreads}+\text{group\_thread\_id}
$$

<h2><span className="num">6.2</span>资源类型：SRV、UAV 与 raw byte access</h2>

HLSL 中常见的 buffer 类型可以按读写能力区分：

<table className="tbl">
    <thead><tr><th>类型</th><th>读写</th><th>典型用途</th><th>xinfer 示例</th></tr></thead>
    <tbody>
      <tr><td><code>StructuredBuffer&lt;T&gt;</code></td><td>只读</td><td>输入 activation、权重、KV cache</td><td><code>A</code>, <code>Q/K/V</code></td></tr>
      <tr><td><code>RWStructuredBuffer&lt;T&gt;</code></td><td>读写</td><td>输出 activation、临时结果</td><td><code>Out</code>, <code>Y</code></td></tr>
      <tr><td><code>ByteAddressBuffer</code></td><td>按 byte 地址只读</td><td>packed f16/int4 权重</td><td><code>linear_f16.hlsl</code></td></tr>
      <tr><td><code>RWByteAddressBuffer</code></td><td>按 byte 地址读写</td><td>原子、压缩输出、特殊布局</td><td>后续高级优化</td></tr>
    </tbody>
  </table>

在 HLSL binding 语法中，`t0` 通常表示 SRV（read-only），`u0` 表示 UAV（read/write），
    `b0` 表示 constant buffer：

```hlsl
StructuredBuffer<float> A : register(t0);      // SRV
ByteAddressBuffer W       : register(t1);      // SRV, raw bytes
RWStructuredBuffer<float> Out : register(u0);  // UAV
cbuffer Params : register(b0) { uint M; uint K; uint N; };
```

本项目使用 FXC / Shader Model 5。SM5 没有像 SM6 那样好用的原生

<code>float16_t</code>

路径，
    但可以用

<code>ByteAddressBuffer.Load</code>

读 32-bit，再用

<code>f16tof32</code>

解包两个 f16。

<h2><span className="num">6.3</span>groupshared memory 与 barrier</h2>

`groupshared` 是同一个 threadgroup 内线程共享的一小块高速内存（AMD 常称 LDS）。
    它常用于归约、tile 缓存、线程间交换临时结果。一个典型模式是：

1. 每个线程计算一个 partial result；
2. 写入 `groupshared` 数组；
3. 调用 `GroupMemoryBarrierWithGroupSync()`；
4. 再做树形归约。

```hlsl
#define TG 64
groupshared float partials[TG];

[numthreads(TG, 1, 1)]
void main(uint3 lid : SV_GroupThreadID) {
    uint t = lid.x;
    partials[t] = local_sum;
    GroupMemoryBarrierWithGroupSync();

    for (uint s = TG / 2; s > 0; s >>= 1) {
        if (t < s) partials[t] += partials[t + s];
        GroupMemoryBarrierWithGroupSync();
    }
}
```

<h2><span className="num">6.4</span>编译 Shader：FXC / SM5 与 DXC / SM6</h2>

HLSL 源码不能直接被 GPU 执行，必须先编译成字节码。Windows 生态中常见两条路线：

<table className="tbl">
    <thead><tr><th>工具</th><th>目标</th><th>特点</th><th>本项目状态</th></tr></thead>
    <tbody>
      <tr><td>FXC</td><td>Shader Model 5</td><td>传统、兼容好；支持 <code>f16tof32</code></td><td>当前使用</td></tr>
      <tr><td>DXC</td><td>Shader Model 6 / DXIL</td><td>现代；wave ops、native f16、更多优化</td><td>未来优化方向</td></tr>
    </tbody>
  </table>

本项目当前的 f16 权重路径使用 SM5 的 `f16tof32`：

```hlsl
uint packed = W.Load(byte_offset);
float w0 = f16tof32(packed & 0xFFFF);
float w1 = f16tof32(packed >> 16);
```

这避免了立即迁移到 DXC，但如果以后要使用 wave intrinsics 或 native half arithmetic，DXC / SM6 会更合适。

<h2><span className="num">6.6</span>资源状态与 barrier：Transition vs UAV</h2>

D3D12 要求你告诉驱动每个资源当前用于什么目的：作为 copy source、copy destination、shader resource，还是 unordered access。
    状态错误会导致调试层报错，严重时可能 device removed。

<table className="tbl">
    <thead><tr><th>Barrier</th><th>作用</th><th>xinfer 中的例子</th></tr></thead>
    <tbody>
      <tr><td>Transition barrier</td><td>把资源从一种状态切到另一种状态</td><td><code>COPY_DEST → NON_PIXEL_SHADER_RESOURCE</code></td></tr>
      <tr><td>UAV barrier</td><td>保证前面的 UAV 写入对后续读写可见</td><td>一个 layer 内多个 dependent dispatch 之间</td></tr>
    </tbody>
  </table>

例如，先把 K/V 写入 cache，再让 attention 读取 cache，就需要保证写入完成并且资源处于可读状态。
    本项目的 `DmlBuffer` 跟踪资源状态，`Executor::uav_barrier()` 用于同一 command list 内的依赖排序。

<h2><span className="num">6.7</span>Root signature：shader 如何看到资源？</h2>

Shader 中写了 `register(t0)`、`register(u0)`，但 GPU 运行时需要知道这些寄存器对应哪些资源。
    Root signature 描述了 shader 可访问资源的布局。本项目的 custom kernel 主要使用：

- **root constants**：少量参数，如 $M,K,N$、count、offset；
- **root descriptors**：直接把 buffer GPU virtual address 绑定给 shader；
- **descriptor tables**：更适合大量资源，本教程前半部分暂不重点使用。

<Callout kind="note" tag="为什么 root descriptors 适合本项目？">

我们的 kernel 参数数量少、资源数量固定，root descriptors 简单直接。更重要的是：它们允许我们把一个 decoder layer
    的多个 dispatch 记录到同一个 command list 中，而不用担心 DirectML binding table 被重复 reset 的问题。

</Callout>

<h2><span className="num">Lab 6</span>写并 dispatch 一个 affine kernel</h2>

Lab 6 对应仓库中的 `shaders/affine.hlsl`。目标是把一个最小 GPU compute path 跑通：

1. 创建输入 buffer 与输出 buffer；
2. 把输入数据上传到 GPU；
3. 编译 HLSL compute shader；
4. 创建 root signature 和 compute pipeline state；
5. 绑定 `count`、SRV、UAV；
6. dispatch；
7. 把结果读回 CPU，验证 $out[i]=in[i]\times2+1$。

这条路径看起来繁琐，但它就是后续 RMSNorm、RoPE、attention、linear kernel 的最小原型。

<QAGroup>

### 思考与练习

<QA question={<><span className="diff easy">基础</span><code>SV_DispatchThreadID</code> 和 <code>SV_GroupThreadID</code> 有什么区别？</>}>

`SV_DispatchThreadID` 是线程在整个 dispatch 网格中的全局索引（= group ID × group 大小 + group 内索引），常用来定位它处理的全局数据元素。`SV_GroupThreadID` 是线程在所属 threadgroup 内的局部索引，常用来索引 groupshared 数组。前者“全局”，后者“组内局部”。

</QA>

<QA question={<><span className="diff easy">基础</span>为什么写入 <code>RWStructuredBuffer</code> 后，后续读它之前通常需要 barrier？</>}>

因为不同 dispatch（或不同 threadgroup）之间没有自动的写后读顺序保证。前一个 dispatch 的 UAV 写可能尚未对后一个 dispatch 的读可见。插入 UAV barrier 确保所有写完成并对后续读可见，避免读到旧数据，保证数据依赖的正确顺序。

</QA>

<QA question={<><span className="diff mid">进阶</span>解释 <code>ByteAddressBuffer</code> 为什么适合 packed f16 权重。</>}>

`ByteAddressBuffer` 以字节偏移寻址，配合 `Load`/`Load2`/`Load4` 读取原始 `uint`。packed f16 是两个 16-bit 权重打包进一个 32-bit uint，用 ByteAddressBuffer 可一次读一个 uint 再用 `f16tof32` 解出两个权重，无需固定 struct stride，也方便做对齐的宽加载（float4）以提升带宽。这正是 xinfer f16 权重路径的做法。

</QA>

<QA question={<><span className="diff mid">进阶</span>如果两个 dispatch 没有数据依赖，是否需要 UAV barrier？为什么？</>}>

不需要。UAV barrier 的作用是强制写后读/写后写的顺序与可见性。若两个 dispatch 操作互不相关的数据（无依赖），插入 barrier 只会阻止它们重叠执行，白白损失并行度。只有当后者读/写前者写过的同一块内存时才需要 barrier。

</QA>

<QA question={<><span className="diff hard">挑战</span>把 affine kernel 改成并行 reduction：计算输入数组的 sum，并说明需要哪些 groupshared 与 barrier。</>}>

方案：每个 threadgroup 处理一段输入。① 每个线程把自己负责的元素（可跨步累加多个）写入 groupshared 数组 `shared[tid]`；② `GroupMemoryBarrierWithGroupSync()`；③ 树形归约：`for (s = groupSize/2; s>0; s>>=1) { if (tid < s) shared[tid]+=shared[tid+s]; GroupMemoryBarrierWithGroupSync(); }`；④ `tid==0` 把 `shared[0]` 用 `InterlockedAdd` 累加到全局输出，或写各组部分和再做二级归约。

需要：一个 `groupshared float shared[GROUP_SIZE]`；每轮归约前后各一次 group sync barrier（写 LDS 后、读相邻槽前都要同步），避免竞态。也可用 `WaveActiveSum` 先在 wave 内归约以减少 barrier 次数。

</QA>

</QAGroup>
