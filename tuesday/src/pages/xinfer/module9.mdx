<h2><span className="num">9.1</span>为什么先搭 Cargo workspace？</h2>

一个 LLM 推理引擎不是单个文件能承载的。即使我们从最小原型开始，也很快会遇到模型数学、GPU backend、权重加载、 tokenizer、runtime、CLI、Xbox host 等不同关注点。如果不在第一天就划清边界，后续每个模块都会互相缠在一起。

因此 Phase 0 的第一步是创建 Cargo workspace，并把未来会变化的部分分成多个 crate：

<h2><span className="num">9.2</span>创建 D3D12 + DirectML device</h2>

第一条真正的 GPU 路径是 device creation。PC 上通常通过 DXGI 枚举 adapter，选择支持 D3D12 的硬件 adapter， 然后创建 `ID3D12Device`。DirectML device 再建立在这个 D3D12 device 之上。

Phase 0 的关键步骤可以概括为：

1. 创建 DXGI factory；
2. 枚举 hardware adapter，跳过 software adapter；
3. 用 `D3D12CreateDevice` 测试并创建 D3D12 device；
4. 创建 direct command queue；
5. 调用 `DMLCreateDevice` 创建 DirectML device；
6. 在 CLI 中打印 adapter 名称，作为 smoke test。

<h2><span className="num">9.3</span>第一个 DirectML GEMM 与第一个 HLSL kernel</h2>

Device 能创建不代表计算正确。Phase 0 还需要两个计算 smoke tests：

- **DirectML GEMM：**验证 DirectML operator create/compile/init/bind/dispatch 全链路。
- **Custom HLSL affine：**验证我们能编译并 dispatch 自己写的 compute shader。

<h3>GEMM 的 CPU reference</h3>

对矩阵 $A\in\mathbb{R}^{M\times K}$、$B\in\mathbb{R}^{K\times N}$，GEMM 输出：

$$
C_{i,j}=\sum_{p=0}^{K-1}A_{i,p}B_{p,j}
$$

GPU 输出必须与 CPU reference 在容差内一致：

$$
\max_{i,j}|C^{\text{gpu}}_{i,j}-C^{\text{cpu}}_{i,j}| < \varepsilon
$$

<h3>Affine kernel</h3>

自写 HLSL kernel 选择一个极简单的函数：

$$
y_i = 2x_i + 1
$$

简单并不意味着没价值。这个 kernel 证明了 root signature、pipeline state、root constants、SRV/UAV 绑定、dispatch、 readback 的最小路径全部可用。后续 RMSNorm、RoPE、SwiGLU、attention 都是从这条路径扩展出来的。

<h2><span className="num">9.4</span>Smoke test 的设计原则</h2>

Phase 0 的测试不应该复杂，但必须能覆盖关键风险：

<table className="tbl">
<thead><tr><th>测试</th><th>覆盖风险</th><th>失败时优先检查</th></tr></thead>
<tbody>
<tr><td><code>xinfer device</code></td><td>D3D12/DML device creation</td><td>驱动、DXGI、DirectML runtime</td></tr>

<Callout kind="note" tag="Phase 0 的核心心态">

不要一开始就写完整 transformer。先让最小 GPU 计算闭环成立：上传 → dispatch → readback → verify。 这个闭环可靠后，后面的复杂 kernel 才有调试基础。

</Callout>

### 思考与练习

<QA question={<><span className="diff easy">基础</span>为什么 Phase 0 不直接实现 transformer，而先实现 affine kernel？</>}>

因为要先打通最小可行管线（device 创建、buffer 分配/上传/下载、kernel 编译/dispatch、fence 同步、结果 readback），并验证它正确。affine kernel（`out=a*in+b`）极简，能在不引入复杂数值的情况下确认整条 GPU 路径可用。若直接上 transformer，一旦出错很难分清是管线 bug 还是模型逻辑 bug。先立“地基”再盖楼。

</QA>

<details className="qa">
<summary><span className="diff easy">基础</span>GEMM 中 $M,K,N$ 分别代表什么？</summary>
<div className="answer">

$C_{M\times N}=A_{M\times K}\cdot B_{K\times N}$。$M$ 是输出行数（如序列长度/batch），$K$ 是被求和消去的内积维（contraction，如 in_features），$N$ 是输出列数（如 out_features）。decode 时 $M=1$（GEMV），prefill 时 $M=$ 序列长度。

</div>
</details>

<QA question={<><span className="diff mid">进阶</span>如果 CPU/GPU GEMM 结果不一致，列出三个可能原因。</>}>

① 行/列主序或转置约定不一致（A 或 B 的 layout、是否 $W^\top$）；② 累加顺序/精度差异（GPU 并行归约 + f16 输入导致与 CPU f32 串行结果有浮点误差——若超出容差才算 bug）；③ 索引/边界错误（越界、stride 算错、threadgroup 覆盖不全导致部分输出未写）。其他还有未同步（读了未完成的 UAV 写）、未初始化输出 buffer 等。

</QA>

<QA question={<><span className="diff mid">进阶</span>解释为什么 readback 是 smoke test 的必要部分。</>}>

因为只有把 GPU 输出拷回 CPU 并和参考值比较，才能证明 kernel 真的算对了。dispatch 不报错不代表结果正确——可能根本没写、写错位置或数值错。readback（经 readback heap + fence 等待）是闭环验证的最后一环，没有它 smoke test 只验证了“能跑”而非“跑对”。

</QA>

<QA question={<><span className="diff hard">挑战</span>把 affine kernel 改成 <code>out[i]=a*in[i]+b</code>，其中 <code>a,b</code> 来自 root constants。</>}>

HLSL：用 root constants 传 a、b（例如 `cbuffer Params : register(b0) { float a; float b; }` 或 32-bit root constants）。kernel：

`uint i = DTid.x; if (i < count) outBuf[i] = a * inBuf[i] + b;`

Rust 侧：root signature 增加一个 root constant 槽（2 个 float），dispatch 前用 `SetComputeRoot32BitConstants` 写入 a、b，再绑定 in/out UAV。相比把 a、b 放进 buffer，root constants 更轻量、无需额外 buffer 与 barrier，适合这种少量标量参数。

</QA>
