<h2><span className="num">3.1</span>自回归生成：一次只预测一个 token</h2>

decoder-only LLM 的生成方式叫做 **autoregressive generation**（自回归生成）：
    模型先根据 prompt 预测下一个 token；然后把这个 token 接到上下文末尾，再预测下一个。这个循环持续到达到最大长度或生成 EOS。

$$
x_{t+1} \sim P_\theta(\cdot \mid x_1,\ldots,x_t)
$$

如果使用 greedy decoding，符号 $\sim$ 可以改写成取最大值：

$$
x_{t+1} = \arg\max_i z_i
$$

其中 $z_i$ 是最后一行 logits 中第 $i$ 个 token 的分数。

<h2><span className="num">3.2</span>Prefill 与 Decode：同一个模型，两种工作负载</h2>

推理通常分成两个阶段：

<table className="tbl">
    <thead><tr><th>阶段</th><th>输入</th><th>目标</th><th>主要工作负载</th></tr></thead>
    <tbody>
      <tr><td>Prefill</td><td>整个 prompt，长度 $S$</td><td>建立上下文表示与 KV cache</td><td>较大的 GEMM，attention 处理 $S&#92;times S$</td></tr>
      <tr><td>Decode</td><td>每次 1 个新 token</td><td>生成下一个 token</td><td>GEMV + 读 KV cache + LM head</td></tr>
    </tbody>
  </table>

在 prefill 中，模型可以一次处理整个 prompt；在 decode 中，每步只处理一个新 token。decode 不能“并行生成未来 token”，
    因为下一个 token 必须依赖刚刚采样出来的 token。

这就是为什么我们报告两个吞吐量：

$$
\text{prefill tok/s} = \frac{\text{prompt tokens}}{\text{prefill seconds}},
    \qquad
    \text{decode tok/s} = \frac{\text{generated tokens}}{\text{decode seconds}}
$$

<h2><span className="num">3.3</span>为什么需要 KV cache？</h2>

在 attention 中，每一层都会计算 key 和 value。假设已经有 $t$ 个历史 token，现在要生成第 $t+1$ 个。
    如果没有 KV cache，每一步都要重新计算所有历史 token 的 $K,V$，导致重复计算。

有 KV cache 后，我们只为新 token 计算一次 $K,V$，然后把它追加到每层的缓存中：

$$
K_{\text{cache}}^{(\ell)}[t] \leftarrow K_{\text{new}}^{(\ell)},\qquad
    V_{\text{cache}}^{(\ell)}[t] \leftarrow V_{\text{new}}^{(\ell)}
$$

attention 时，新 query 会读取缓存中从 0 到 $t$ 的所有 keys/values：

$$
\operatorname{Attn}(q_t,K_{\le t},V_{\le t})
      = \operatorname{softmax}\left(\frac{q_tK_{\le t}^{\top}}{\sqrt{d}}\right)V_{\le t}
$$

<Callout kind="def" tag="KV cache 的本质">

KV cache 是“用显存换计算”：缓存过去 token 的 K/V，避免每一步重新运行历史 token 的 QKV projection。
    它不改变模型数学输出，只改变计算方式。

</Callout>

<h3>KV cache 的显存公式</h3>

对 decoder-only 模型，KV cache 大小大约是：

$$
\operatorname{bytes}_{KV}
      = 2 \cdot L \cdot T \cdot n_{\text{kv}} \cdot d_{\text{head}} \cdot \operatorname{bytes}(\text{dtype})
$$

其中 2 表示 K 和 V，$L$ 是层数，$T$ 是缓存容量，$n_{\text{kv}}$ 是 KV head 数。
    GQA 的价值在这里非常明显：如果 $n_{\text{kv}}$ 从 14 降到 2，KV cache 也按比例下降。

<h2><span className="num">3.4</span>采样：从 logits 到下一个 token</h2>

模型给出 logits $z$，采样器决定最终选哪个 token。不同采样策略会让输出更确定或更多样。

<h3>Greedy / Argmax</h3>

$$
x_{t+1} = \arg\max_i z_i
$$

Greedy 最稳定、可复现，但有时会过于保守。本项目对 greedy 做了 GPU argmax 优化：只读回一个 `u32` token id，
    不读回整个词表 logits。

<h3>Temperature</h3>

temperature $ \tau $ 会缩放 logits：

$$
p_i = \operatorname{softmax}\left(\frac{z_i}{\tau}\right)
$$

$\tau$ 越小，分布越尖锐；$\tau$ 越大，低概率 token 更有机会被选中。

<h3>Top-k 与 Top-p</h3>

Top-k 只保留概率最高的 $k$ 个 token。Top-p（nucleus sampling）先按概率从高到低排序，
    只保留累积概率达到 $p$ 的最小集合：

$$
\mathcal{S}_p=\min\left\{S:\sum_{i\in S}p_i\ge p\right\}
$$

<h2><span className="num">3.5</span>Tokenizer、chat template 与 special tokens</h2>

模型并不直接处理 Unicode 字符串，而是处理 token id。Tokenizer 负责双向转换：

$$
\text{text}\;\xrightarrow{\text{tokenizer}}\;[x_1,\ldots,x_S]
    \qquad
    [y_1,\ldots,y_T]\;\xrightarrow{\text{decode}}\;\text{text}
$$

对 instruct model，通常还要套上 chat template。例如 Qwen 的对话格式类似：

```text
<|im_start|>system
You are a helpful assistant.<|im_end|>
<|im_start|>user
What is the capital of France?<|im_end|>
<|im_start|>assistant
```

EOS token 表示模型认为回答结束。生成循环一般在满足以下条件之一时停止：

- 生成了 EOS token；
- 达到 `max_new_tokens`；
- 外部系统要求中断（例如 UI 停止按钮）。

<h2><span className="num">Lab 3</span>纯 CPU 生成循环：KV cache + samplers</h2>

这个实验不需要 GPU。你将用一个 tiny 模型或假 logits 实现完整生成循环，重点是控制流而不是模型规模。

```python
def generate(prompt_tokens, max_new_tokens):
    cache = empty_kv_cache()
    logits = model_prefill(prompt_tokens, cache)
    next_token = sample(logits[-1])

    output = []
    for _ in range(max_new_tokens):
        if next_token == EOS:
            break
        output.append(next_token)
        logits = model_decode_one(next_token, cache)
        next_token = sample(logits[-1])
    return output
```

### 思考与练习

<QA question={<><span className="diff easy">基础</span>解释 prefill 和 decode 的区别。为什么 decode 不能一次并行生成 10 个未来 token？</>}>

prefill 一次性处理整个 prompt（多个 token 并行），建立上下文表示和 KV cache；decode 每次只处理一个新 token。decode 不能一次并行生成 10 个未来 token，是因为自回归依赖：第 $t+2$ 个 token 的输入依赖第 $t+1$ 个 token 的采样结果，而后者必须先生成。换句话说，未来 token 之间存在串行因果链，无法在不知道前一个 token 的情况下计算后一个。（speculative decoding 是用小模型“猜”多个 token 再验证，属于另一种思路。）

</QA>

<QA question={<><span className="diff easy">基础</span>如果 prompt 有 128 个 token，生成 32 个 token，KV cache 最终 len 是多少？</>}>

$128+32=160$。prefill 写入 128 个槽位，随后每 decode 一个 token 追加一个槽位，共 32 个。注意为避免溢出，cache capacity 至少要预留 160。

</QA>

<details className="qa">
<summary><span className="diff mid">进阶</span>推导 KV cache 显存公式，并代入 $L=24,T=1024,n_{\text{kv}}=2,d=64,\text{dtype}=f32$。</summary>
<div className="answer">

每层每个 token 要存 K 和 V，各 $n_{\text{kv}}\times d$ 个元素，所以：$\text{bytes}=2\cdot L\cdot T\cdot n_{\text{kv}}\cdot d\cdot \text{bytes(dtype)}$。

代入：$2\times24\times1024\times2\times64\times4=2\times24\times1024\times512=25{,}165{,}824$ 字节 $\approx 24$ MiB。可见上下文 $T$ 越长，KV cache 线性增长；GQA 让 $n_{\text{kv}}=2$ 而非 14，已经把它压到 1/7。

</div>
</details>

<details className="qa">
<summary><span className="diff mid">进阶</span>给定 logits $[1,2,3]$，分别计算 temperature $1.0$ 和 $0.5$ 下 softmax 分布的变化趋势。</summary>
<div className="answer">

$\tau=1.0$：softmax([1,2,3]) ≈ [0.090, 0.245, 0.665]。

$\tau=0.5$：先把 logits 除以 0.5 得到 [2,4,6]，softmax([2,4,6]) ≈ [0.016, 0.117, 0.867]。

趋势：温度变小，分布变得更尖锐，最大 logit 对应 token 的概率更突出（0.665 → 0.867），低概率 token 被进一步压制。温度越大则相反，分布更平、更随机。

</div>
</details>

<details className="qa">
<summary><span className="diff hard">挑战</span>实现 top-p sampling，并写一个测试证明保留集合的累积概率至少为 $p$。</summary>
<div className="answer">

实现步骤：① 对概率从高到低排序；② 累加，直到累积和首次 $\ge p$，取到该位置为止的最小集合 $\mathcal{S}_p$；③ 在 $\mathcal{S}_p$ 内重新归一化后按概率采样。

测试思路：构造若干随机分布，对每个分布运行 top-p，断言 `sum(probs in S_p) >= p - 1e-6`（留浮点容差）。由于算法是“累加到首次达到 $p$ 才停止”，集合累积概率必然 $\ge p$，同时它是满足该条件的最小集合（去掉最后一个元素就 $<p$）。

</div>
