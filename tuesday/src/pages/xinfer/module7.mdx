<h2><span className="num">7.1</span>Qwen2 decoder 架构与 config.json</h2>

到目前为止，我们讨论的是抽象的 decoder-only transformer。现在要选一个真实模型作为教学目标。
    本项目选择 **Qwen2 / Qwen2.5**，原因是：架构现代、公开权重可用、模型尺寸覆盖 0.5B 到更大规模，
    并且它包含许多真实系统会遇到的结构：GQA、RoPE、RMSNorm、SwiGLU、QKV bias、tied embeddings。

以本仓库实际下载的 `Qwen2.5-0.5B-Instruct` 为例，关键配置如下：

由这些字段可以推导 attention head 维度：

$$
d_{\text{head}} = \frac{H}{n_{\text{heads}}}
  = \frac{896}{14}=64
$$

因此 Q 的 shape 是 $[S,14,64]$，而 K/V 因为 GQA 只有 2 个 KV head，shape 是 $[S,2,64]$。

<h2><span className="num">7.2</span>Qwen2 的关键结构</h2>

Qwen2 的 decoder layer 与 Module 2 的公式一致，但有一些必须在实现中准确处理的细节。

<h3>RMSNorm</h3>

Qwen2 使用 RMSNorm，稳定项由 `rms_norm_eps` 给出：

$$
\operatorname{RMSNorm}(x)_i
  = \frac{x_i}{\sqrt{\frac{1}{H}\sum_j x_j^2+\varepsilon}}g_i
$$

<h3>RoPE 与 rope_theta</h3>

Qwen2 对 Q/K 使用 RoPE。配置中的 `rope_theta` 是频率基底：

$$
\omega_i = \theta_{\text{base}}^{-2i/d_{\text{head}}},
\qquad
\phi_{p,i}=p\omega_i
$$

<h3>GQA head counts</h3>

Qwen2.5-0.5B 有 14 个 query heads，但只有 2 个 KV heads，因此每个 KV head 被 7 个 query head 共享：

$$
\text{group size} = \frac{14}{2}=7
$$

<h3>QKV bias</h3>

很多 Llama 系模型的 Q/K/V projection 不带 bias，但 Qwen2 带 QKV bias：

$$
Q=XW_Q^\top+b_Q,\quad K=XW_K^\top+b_K,\quad V=XW_V^\top+b_V
$$

因此 loader 和 model assembly 必须加载 `q_proj.bias`、`k_proj.bias`、`v_proj.bias`。

<h3>SwiGLU</h3>

MLP 使用 gate/up/down 三个矩阵：

$$
\operatorname{MLP}(x)
  = \left(\operatorname{SiLU}(xW_g^\top)\odot xW_u^\top\right)W_d^\top
$$

<h2><span className="num">7.3</span>Tied vs untied LM head 与词表规模</h2>

如果 `tie_word_embeddings=true`，LM head 权重与 embedding table 共享：

$$
W_{\text{lm}} = E
$$

这能减少参数量和显存。对 Qwen2.5-0.5B：

$$
|V|\cdot H = 151936 \times 896 \approx 1.36\times 10^8
$$

如果用 f16 存储，这张表约为：

$$
151936 \cdot 896 \cdot 2 \approx 272\text{ MB}
$$

<Callout kind="warn" tag="性能提醒">

大词表不仅占内存，也影响每个 token 的 LM head 计算和 logits 读回。xinfer 后来的 GPU argmax 优化正是为了避免每步读回
    $[151936]$ 个 logits。

</Callout>

<h2><span className="num">7.4</span>权重格式：safetensors、dtype 与 memory mapping</h2>

<h2><span className="num">7.5</span>选择一个合适大小：为什么从 0.5B 开始？</h2>

教学模型要满足两个条件：足够真实，又能在普通开发机上快速迭代。Qwen2.5-0.5B 有 24 层、真实 tokenizer、真实 Qwen2 结构，
    但权重规模仍然可控。xinfer 在 f16 权重后，GPU resident weights 约 942 MiB。

随模型规模增长，主要增长项包括：

- 层数 $L$：所有 per-layer 权重、KV cache 都随 $L$ 线性增长；
- hidden size $H$：线性层权重通常随 $H^2$ 或 $H\cdot I$ 增长；
- intermediate size $I$：MLP gate/up/down 权重随 $H\cdot I$ 增长；
- 词表大小 $|V|$：embedding / LM head 随 $|V|\cdot H$ 增长；
- 上下文长度 $T$：KV cache 随 $T$ 线性增长。

$$
\operatorname{shape}(W_Q) = [H,H],\qquad

$$

### 思考与练习

<details className="qa">
<summary><span className="diff easy">基础</span>从 Qwen2.5-0.5B 的 config 推导 $d_{\text{head}}$。</summary>
<div className="answer">

$d_{\text{head}}=\text{hidden\_size}/\text{num\_attention\_heads}=896/14=64$。这也是 config 中 head_dim 的值，K/V 每个 head 同样是 64。

</div>
</details>

<QA question={<><span className="diff easy">基础</span>解释 <code>tie_word_embeddings=true</code> 的含义。</>}>

表示输入 embedding 矩阵与输出 LM head 的权重共享同一份参数（LM head = embedding 的转置）。好处是省下一份 $\text{vocab}\times\text{hidden}=151936\times896$ 的大矩阵（f16 约 260 MiB），并提供正则化效果。Qwen2.5-0.5B 即采用 tied embeddings，因此 loader 不需要单独的 `lm_head.weight`。

</QA>

<details className="qa">
<summary><span className="diff mid">进阶</span>为什么 Qwen2 的 K/V projection 输出维度是 $2\times64=128$，而 Q projection 是 $14\times64=896$？</summary>
<div className="answer">

因为用了 GQA：query 有 14 个 head（$14\times64=896$），但 K/V 只有 2 个 KV head（$2\times64=128$）。多个 query head 共享同一对 K/V，从而把 KV 投影和 KV cache 缩到 1/7。这是在质量与显存/带宽之间的折中。

</div>
</details>

<QA question={<><span className="diff mid">进阶</span>估算 embedding table 的 f16 显存大小。</>}>

$\text{vocab}\times\text{hidden}\times 2\text{ bytes}=151936\times896\times2\approx 272{,}269{,}312$ 字节 $\approx 260$ MiB。由于 tie_word_embeddings，这一份同时充当 LM head，不必再翻倍。

</QA>

<QA question={<><span className="diff hard">挑战</span>写一个脚本列出 safetensors 中所有 tensor，并按总字节数排序前 10。</>}>

safetensors 文件头是一段 JSON，记录每个 tensor 的 dtype、shape 和 data_offsets。脚本思路（Python，`safetensors` 库）：

`from safetensors import safe_open`；用 `safe_open(path,'pt')` 打开，遍历 `f.keys()`，对每个 key 取 `t=f.get_slice(k)` 拿 shape/dtype，按 $\prod\text{shape}\times\text{bytes(dtype)}$ 计算字节数；收集成列表后 `sorted(..., key=size, reverse=True)[:10]` 打印。预期前列是 embedding（≈260 MiB）、各层 MLP 的 gate/up/down 与 attention 的 q/o 投影。也可直接解析文件头 JSON 的 data_offsets 差值得到字节数，无需加载数据。

</QA>
