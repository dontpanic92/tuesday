<h2><span className="num">5.1</span>大图景：所有 API 都在表达同一件事</h2>

不管你使用 CUDA、Vulkan、Metal、OpenCL 还是 Direct3D 12，GPU compute 的核心动作都类似：
    准备 buffer，把 kernel 编译成 GPU 能执行的形式，绑定资源，向 command queue 提交 dispatch，
    最后通过 fence / event 等机制等待完成。

<h2><span className="num">5.2</span>CUDA：深度学习的事实标准</h2>

CUDA 是 NVIDIA 的专有 GPU compute 平台。它之所以成为深度学习事实标准，不只是因为 kernel 语言好用，
    更因为围绕它有完整生态：cuBLAS、cuDNN、NCCL、TensorRT、Nsight、成熟的 PyTorch/TensorFlow 后端。

<table className="tbl">
    <thead><tr><th>优点</th><th>代价</th></tr></thead>
    <tbody>
      <tr><td>生态成熟，资料丰富</td><td>绑定 NVIDIA GPU</td></tr>
      <tr><td>cuBLAS/cuDNN 性能极强</td><td>Windows/Xbox/AMD/Apple 路径不适用</td></tr>
      <tr><td>调试和 profiling 工具完善</td><td>跨平台抽象需要额外层</td></tr>
      <tr><td>很多论文和 kernel 以 CUDA 为参考实现</td><td>教学上容易把 matmul 直接交给库，隐藏细节</td></tr>
    </tbody>
  </table>

<Callout kind="note" tag="本教程视角">

如果教程目标是“最快跑起来”，CUDA + cuBLAS 是合理选择；如果目标是理解 GPU compute backend，
    并兼顾 Windows / Xbox，则 CUDA 不是合适的唯一后端。

</Callout>

<h2><span className="num">5.3</span>Vulkan compute 与 SPIR-V</h2>

Vulkan 是跨厂商、跨平台的低层图形/计算 API。它的 shader 通常编译成 SPIR-V 中间表示。
    Vulkan 的优势是可移植性强：Windows、Linux、Android，NVIDIA/AMD/Intel 都支持。但它也非常显式：
    设备、内存、descriptor set、pipeline、barrier、queue family 都需要程序员管理。

Vulkan 适合构建跨平台底层 backend，但教学门槛比 D3D12/HLSL 更高；而且 Xbox 原生路径不是 Vulkan。

<h2><span className="num">5.4</span>Metal 与 OpenCL：作为背景</h2>

Metal 是 Apple 平台的现代 GPU API。它在 macOS / iOS 上体验很好，工具链统一，但不适用于 Windows / Xbox。
    如果教程目标是 Apple 生态，Metal 是自然选择；但本项目的主平台是 Windows PC 与 Xbox GDK。

OpenCL 曾经是通用计算跨平台方案，但生态和性能调优体验不如 CUDA / Vulkan / Metal / D3D12 现代路线。
    了解 OpenCL 有助于理解 GPU compute 历史，但本教程不选它作为主线。

<h2><span className="num">5.5</span>Direct3D 12 + DirectML：本项目的选择</h2>

本项目选择 Direct3D 12 + DirectML，有几个现实原因：

- **Windows 原生：**目标机器是 Windows PC，D3D12 是第一等公民。
- **Xbox 路径：**Xbox GDK 使用 D3D12 风格设备创建；host 可以把设备注入 Rust core。
- **HLSL 教学友好：**HLSL compute shader 语法直观，适合讲 threadgroup、SRV/UAV、barrier。
- **DirectML 可作为 operator API：**我们可以研究它的生命周期，也能在遇到问题时切换到自写 kernel。

<Callout kind="warn" tag="真实案例：为什么不能盲信库算子">

在本项目机器的 RDNA4 GPU 上，DirectML GEMM 在某些 shape 会导致 device removed。
    因此 matmul 最终改成自写 HLSL kernel。这个案例正好说明：理解 backend 不是可选项。

</Callout>

<h2><span className="num">5.6</span>可移植性策略：为什么要建 HAL？</h2>

如果模型代码里到处直接调用 `ID3D12Resource`、root signature、descriptor heap，
    将来换 backend 会非常痛苦。HAL（Hardware Abstraction Layer）把模型需要的能力抽象成更稳定的接口：

$$
\text{Model code} \rightarrow \text{Device / Buffer / Kernel traits}
\rightarrow \text{Backend implementation}
$$

在 xinfer 中，`xinfer-backend` 定义了抽象边界，`xinfer-dml` 实现 D3D12/DirectML backend。
    Xbox 支持也依赖这个思想：console host 创建设备，然后把 device 指针注入 Rust core；模型和 runtime 不需要知道设备来自 DXGI 还是 GDK。

      <tr><td>Metal</td><td>Apple 平台最佳体验</td><td>不覆盖 Windows/Xbox</td></tr>

教程选择 D3D12 + DirectML，不是因为它“绝对最好”，而是因为它最符合本项目目标：
    Windows 可运行、Xbox 可迁移、HLSL 可教学、底层足够透明。

### 思考与练习

<QA question={<><span className="diff easy">基础</span>列出所有 GPU compute API 都需要表达的 5 个共同概念。</>}>

① 设备/队列（device & command queue，提交工作）；② 缓冲区/资源（buffer/memory，存数据）；③ kernel/shader（要执行的程序）；④ 资源绑定（descriptor/binding，告诉 kernel 用哪些 buffer）；⑤ dispatch + 同步（启动 N 个线程组并用 fence/barrier 同步）。无论 CUDA、D3D12、Vulkan、Metal，都要表达这五件事。

</QA>

<QA question={<><span className="diff easy">基础</span>为什么 CUDA 在深度学习中很强，但不适合作为本教程唯一 backend？</>}>

CUDA 生态成熟、kernel 库丰富，但它只支持 NVIDIA GPU。本教程目标是 Windows PC + Xbox，且要在 AMD（RX 9070 XT）上跑，还要走 Xbox GDK 路线——这些都不支持 CUDA。DirectML/D3D12 才能覆盖跨厂商 GPU 与主机平台，所以选它作为可移植 backend。

</QA>

<QA question={<><span className="diff mid">进阶</span>解释 descriptor / resource binding 在 D3D12 或 Vulkan 中解决了什么问题。</>}>

它解决“shader 如何知道该访问哪块 GPU 内存”的问题。shader 里只写逻辑寄存器槽（如 `register(u0)`），descriptor 把这些槽映射到具体的 GPU 资源（buffer/texture 的地址、格式、范围）。这层间接让同一个 shader 可在不同帧绑定不同资源，并让驱动统一管理资源生命周期与访问权限。xinfer 用 root descriptor 直接绑定 buffer 地址，省去描述符堆重绑。

</QA>

<QA question={<><span className="diff mid">进阶</span>如果模型代码直接依赖 D3D12 COM 类型，会怎样影响未来移植？</>}>

模型层会和 Windows/D3D12 紧耦合：想换 Vulkan/Metal 或上别的平台就得改 Qwen 前向逻辑本身，而不仅是 backend。这违背关注点分离。正确做法是把 D3D12 细节藏在 backend trait（Device/Buffer/Kernel）后面，让 `xinfer-model` 只依赖抽象接口，移植时只替换 `xinfer-dml` 这一层。

</QA>

<QA question={<><span className="diff hard">挑战</span>为 xinfer 设计一个假想 Vulkan backend 的 crate 边界，说明哪些 trait 可以复用。</>}>

新增 `xinfer-vk` crate，实现 `xinfer-backend` 中已有的同一组 trait：`Device`（alloc/upload/download/创建 kernel）、`Buffer`（GPU 内存句柄+状态）、kernel dispatch / executor（命令缓冲 + fence）。Vulkan 概念一一对应：VkDevice/VkQueue↔Device，VkBuffer+VkDeviceMemory↔Buffer，VkPipeline(compute)+descriptor set↔Kernel/绑定，VkCommandBuffer+VkFence↔Executor。可完全复用的是上层：`xinfer-core`（tensor/config/CPU 参考）、`xinfer-model`（Qwen 结构）、`xinfer-runtime`（KV cache/采样/decode 循环）、`xinfer-loader`、`xinfer-tokenizer`——它们都只依赖 trait，不依赖具体 API。HLSL shader 需重编为 SPIR-V，是少数需要改的部分。

</QA>
