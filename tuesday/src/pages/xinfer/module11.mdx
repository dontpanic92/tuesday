<h2><span className="num">11.1</span>CPU reference first：正确性 oracle</h2>

写 GPU kernel 最危险的地方是：它可能“看起来能跑”，但数值悄悄错了。调试 GPU 输出比调试 CPU 函数困难得多， 因此每个 kernel 都要先有一个简单、清晰、慢但可信的 CPU reference。

在 xinfer 中，CPU reference 放在 `xinfer_core::cpu`，采用统一 layout：

- hidden states：`[seq, hidden]`
- Q：`[seq_q, n_heads, head_dim]`
- K/V：`[seq_k, n_kv_heads, head_dim]`

每个 GPU kernel 的测试都遵循同一个结构：

$$
\text{input} \xrightarrow{\text{CPU reference}} y_{\text{cpu}},
    \qquad
    \text{input} \xrightarrow{\text{GPU kernel}} y_{\text{gpu}},
    \qquad
    \lVert y_{\text{gpu}}-y_{\text{cpu}}\rVert_\infty < \varepsilon
$$

<h3>RMSNorm kernel</h3>

CPU 公式是：

$$
y_i = \frac{x_i}{\sqrt{\frac{1}{H}\sum_j x_j^2+\varepsilon}}g_i
$$

GPU 上，最直接的版本可以一个 thread 处理一整行；优化版则让一个 threadgroup 合作归约。无论哪种版本， 测试输入和输出 shape 都是 `[seq, hidden]`。

<h3>RoPE kernel</h3>

RoPE 对 Q/K 的每个 head 做旋转。GPT-NeoX / Qwen2 约定把 head_dim 的前半和后半配对：

$$
a' = a\cos\phi - b\sin\phi,\qquad
    b' = b\cos\phi + a\sin\phi
$$

它是 in-place kernel：输入 buffer 被直接改写。一个常见测试是验证 L2 norm 不变。

<h3>GQA attention kernel</h3>

注意力 kernel 要处理 `seq_q` 与 `seq_k` 可能不同的情况，这为 KV cache 做准备。 对 query row $t_q$，可见 key 数是：

$$
\text{key\_count} = \min(q\_\text{pos\_base}+t_q+1,\; \text{seq\_k})
$$

在 Phase 2 中，attention kernel 的重点不是最快，而是把 GQA mapping、causal mask、softmax 稳定性都做对。

<h2><span className="num">11.3</span>Matmul 与 transposed-weight linear</h2>

HuggingFace 权重通常按 `[out_features, in_features]` 存储，因此线性层是：

$$
Y = XW^\top
$$

如果 $X\in\mathbb{R}^{M\times K}$，$W\in\mathbb{R}^{N\times K}$，则：

$$
Y_{i,j}=\sum_{p=0}^{K-1}X_{i,p}W_{j,p},
    \qquad
    Y\in\mathbb{R}^{M\times N}
$$

Phase 2 先用 DirectML GEMM / transposed-B 路径验证 linear 层。但真实项目后来发现某些 RDNA4 shape 会触发 DirectML GEMM device removed，因此 Phase 3 之后 matmul 迁移到 custom HLSL。教学上这很好：Phase 2 让你先理解标准 GEMM， 后续模块再学习为什么要定制 kernel。

<h2><span className="num">11.4</span>数值 parity：为什么很少 bit-exact？</h2>

CPU 与 GPU 做同一个数学计算，结果通常不会逐 bit 完全相同。原因包括：

- 浮点加法不满足结合律：$(a+b)+c$ 与 $a+(b+c)$ 可能不同；
- GPU 归约顺序可能与 CPU loop 不同；
- f16 / bf16 权重会引入舍入；
- softmax / exp 对微小误差敏感；
- 不同硬件可能使用不同 fused multiply-add 行为。

因此测试通常设定容差：

$$
\max_i |y^{gpu}_i-y^{cpu}_i| < \varepsilon
$$

选择 $\varepsilon$ 时要看 dtype 和算子。纯 f32 的 elementwise add 可以很小；attention / matmul / f16 权重通常需要稍大一些。 除了数值差，也可以检查 argmax 是否稳定，因为生成时最终常常只关心最大 logit 对应的 token。

### 思考与练习

<QA question={<><span className="diff easy">基础</span>为什么 CPU reference 必须比 GPU kernel 更简单？</>}>

因为 CPU 参考是“正确性的真值来源”。它越简单（直白的三重循环、串行、f32），越不容易自身出错，也越容易肉眼核对。GPU kernel 为性能引入并行、分块、向量化、f16，复杂且易错；用一个简单可信的 CPU 实现做 parity 比较，才能把错误隔离在 GPU 侧。若两边都复杂，出错时无法判断谁是对的。

</QA>

<QA question={<><span className="diff easy">基础</span>写出 transposed-weight linear 的 shape 关系。</>}>

权重以 $W\in\mathbb{R}^{N\times K}$（out×in）存储，计算 $Y=XW^\top$：$X\in\mathbb{R}^{M\times K}$，$W^\top\in\mathbb{R}^{K\times N}$，$Y\in\mathbb{R}^{M\times N}$。即对每个输出元素 $Y_{m,n}=\sum_k X_{m,k}W_{n,k}$——注意 W 的两个下标都按行存，正好让同一输出列读取 W 的连续一行，利于 coalescing。

</QA>

<QA question={<><span className="diff mid">进阶</span>解释 GQA 中 query head 到 KV head 的映射公式。</>}>

设 group size $g=n_{\text{heads}}/n_{\text{kv}}$。query head $h$ 使用的 KV head 为 $\text{kv}(h)=\lfloor h/g\rfloor$。Qwen2.5-0.5B 中 $g=14/2=7$：head 0–6 → KV 0，head 7–13 → KV 1。attention 计算时 Q 用 $h$ 索引、K/V 用 $\lfloor h/g\rfloor$ 索引。

</QA>

<QA question={<><span className="diff mid">进阶</span>为什么 attention softmax 必须减去最大 score？</>}>

数值稳定性。softmax 要算 $e^{s_i}$，若 $s_i$ 较大，$e^{s_i}$ 会溢出成 inf。减去最大值 $m=\max_i s_i$ 后算 $e^{s_i-m}$，最大项变成 $e^0=1$，其余 $\le1$，不会上溢；由于分子分母同乘 $e^{-m}$，结果数学上不变。这就是“safe softmax”，flash attention 的 online softmax 也据此维护 running max 并 rescale。

</QA>

<QA question={<><span className="diff hard">挑战</span>构造一个小输入，手算 causal attention 输出，并用它作为单元测试。</>}>

取 1 head，$d=1$，两个 token。令 $Q=[q_0,q_1]$，$K=[k_0,k_1]$，$V=[v_0,v_1]$，缩放 $1/\sqrt{1}=1$。

token 0（只能看自己）：score=$q_0k_0$，softmax 单元素=1，输出 $=v_0$。

token 1（看 0,1）：scores $s_0=q_1k_0,\ s_1=q_1k_1$；$w_0=\frac{e^{s_0-m}}{e^{s_0-m}+e^{s_1-m}}$，$w_1=1-w_0$；输出 $=w_0v_0+w_1v_1$。

代入具体数（如 $q=[1,1],k=[1,2],v=[3,5]$）：token1 scores=[1,2]，$w=\text{softmax}([1,2])\approx[0.269,0.731]$，输出 $\approx0.269\cdot3+0.731\cdot5\approx4.46$。把这些常数硬编码进测试，断言 kernel 输出在容差内匹配 [3.0, 4.46]，并验证 token0 不会看到 token1（causal）。

</QA>
