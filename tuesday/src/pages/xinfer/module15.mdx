<h2><span className="num">15.1</span>Benchmark 要可重复</h2>

如果每次测量的 prompt、采样参数、模型、运行模式都不同，数字就没有可比性。最小 benchmark 配置应该固定：

- 模型路径，例如`models\\Qwen2.5-0.5B-Instruct`；
- prompt；
- `max_new_tokens`；
- 采样策略：greedy 或固定 temperature/top-k/top-p；
- 随机种子；
- resident 或 streaming 模式。

```bash
xinfer generate --model models\Qwen2.5-0.5B-Instruct \
  --prompt "Count from one to twenty." \
  --max-tokens 60 --no-chat --seed 1
```

Runtime 应该分别记录：

$$
\text{prefill tok/s} = \frac{N_{\text{prompt}}}{t_{\text{prefill}}},
\qquad
\text{decode tok/s} = \frac{N_{\text{generated}}}{t_{\text{decode}}}
$$

<h2><span className="num">15.2</span>内存报告：不要只看 tok/s</h2>

LLM 推理不仅受算力影响，也受显存容量和带宽影响。因此 benchmark 至少要报告：

- resident GPU weights 大小；
- KV cache 大小；
- 是否 streaming，以及 resident layer slots 数；
- 每 token 是否 readback 完整 logits；
- 权重 dtype：f32/f16/int8/int4。

KV cache 显存公式：

$$
\operatorname{bytes}_{KV}
=2LTn_{\text{kv}}d_{\text{head}}\operatorname{bytes(dtype)}
$$

这意味着更长上下文 $T$、更多层 $L$、更多 KV heads 都会线性增加 KV cache。

<h2><span className="num">15.3</span>找瓶颈：submission、bandwidth、readback</h2>

xinfer 的优化历史说明：真正的瓶颈不一定是你最先怀疑的地方。我们最初以为 matmul kernel 是最大问题； 实际上前三个大瓶颈分别是：过多 GPU submission、f32 权重带宽、完整 logits readback。

<h2><span className="num">15.4</span>CPU wall-clock vs GPU timestamp</h2>

CPU wall-clock 测到的是“从用户角度看花了多久”，包含 CPU 准备、command submission、GPU 执行、fence wait、readback 等。 GPU timestamp 测到的是“GPU 命令区间本身花了多久”。两者都重要，但回答的问题不同。

<table className="tbl">
<thead><tr><th>计时方式</th><th>回答的问题</th><th>包含</th><th>不包含</th></tr></thead>
<tbody>
<tr><td>CPU wall-clock</td><td>用户实际等待多久？</td><td>提交、等待、readback、CPU 逻辑</td><td>难以分辨 GPU 本体耗时</td></tr>
<tr><td>GPU timestamp</td><td>某段 GPU 工作花多久？</td><td>GPU command 执行区间</td><td>CPU 调度、readback、等待开销</td></tr>
</tbody>
</table>

一个成熟 benchmark 会同时看二者。例如 Tier 1 优化主要改善 CPU/GPU submission；Tier 3 优化主要改善 GPU kernel 内存访问； Tier 6 优化主要减少 readback。

<h2><span className="num">15.5</span>Regression gate：正确性永远是门槛</h2>

性能优化必须以正确性为前提。每次优化都要运行 parity tests：

$$
\max_i |y_i^{gpu}-y_i^{cpu}| < \varepsilon
$$

对生成系统，还要检查：

- KV cache incremental decode 是否等于 full forward；
- streaming 是否等于 resident；
- GPU argmax 是否等于 CPU argmax；
- 真实 prompt 是否仍然输出合理答案。

<Callout kind="warn" tag="不要只追 tok/s">

一个快但错的 kernel 没有教学价值，也没有工程价值。优化报告必须同时写出 correctness gate 和性能数据。

</Callout>

### 思考与练习

<QA question={<><span className="diff easy">基础</span>为什么 prefill tok/s 和 decode tok/s 不能混在一起报告？</>}>

因为两者性能特征完全不同。prefill 并行处理多个 token，算术强度高、接近计算受限，tok/s 很高；decode 每次一个 token（GEMV），带宽受限、还有固定提交开销，tok/s 低得多。把它们平均会同时掩盖 prefill 的高吞吐和 decode 的真实瓶颈，对优化没有指导意义。应分别报告（如 xinfer 的`GenerateStats`拆成 prefill ms 与 decode tok/s）。

</QA>

<QA question={<><span className="diff easy">基础</span>列出一个可复现 benchmark 至少需要固定的 4 个参数。</>}>

① 模型与精度（如 Qwen2.5-0.5B、f16 权重）；② prompt 长度 / 生成 token 数；③ 采样设置（greedy 还是带温度，影响是否走 GPU argmax fast path）；④ 硬件与驱动 + 是否预热（warmup 后取稳定态、排除首次编译/上传）。其他还有 batch size、上下文长度、是否 streaming。固定这些才能让前后对比有意义。

</QA>

<QA question={<><span className="diff mid">进阶</span>如果优化后 GPU timestamp 变快但 wall-clock 没变，可能说明什么？</>}>

说明瓶颈不在 GPU kernel 执行本身，而在 GPU 之外：可能是 CPU 侧提交开销、驱动/同步、每 token 的 readback 与 CPU 采样、或 CPU/GPU 没有重叠（GPU 算得快但大部分时间在等 CPU）。此时该优化命令提交批处理、减少 readback、让 CPU/GPU 重叠，而不是继续抠 kernel。这正是 T1 批处理（530→24 submits/token）带来最大单项提升的原因。

</QA>

<QA question={<><span className="diff mid">进阶</span>为什么 GPU argmax 优化主要改善 readback，而不是 matmul 本身？</>}>

因为 LM head matmul 仍要算出全部 151936 个 logits，计算量不变；GPU argmax 只是改变“取结果”的方式——在 GPU 上归约出最大下标，把 readback 从 608 KB 降到 4 字节。它省的是 PCIe 回传与 CPU 端处理时间，不是 matmul 的 FLOPs。所以它归类为 readback/数据传输优化，而非计算优化。

</QA>

<QA question={<><span className="diff hard">挑战</span>设计一个实验区分“submission overhead”与“kernel bandwidth bottleneck”。</>}>

思路：分别独立地放大两个变量看 tok/s 如何响应。

<span className="label">测 submission overhead：</span>保持 kernel 工作量不变，改变每 token 的提交次数（如对比 per-op flush 与 per-layer/per-token 批处理）。若 tok/s 随提交次数下降而显著上升，说明此前受提交开销支配。也可插入大量极小的空 dispatch，测纯提交成本。

<span className="label">测 bandwidth bottleneck：</span>保持提交结构不变，改变权重字节数（f32→f16 减半带宽，或人为加大/缩小矩阵）。若耗时随读取字节数近似线性变化、且 GPU timestamp 与理论带宽吻合，则是带宽受限。

结合 GPU timestamp（测 kernel 执行）与 wall-clock（含提交）两套计时：若 wall≫timestamp 之和 → 提交/同步主导；若 timestamp 随字节数线性 → 带宽主导。xinfer 的实测正是先靠 T1 砍提交（3.6→8.5），再靠 T2 f16 砍带宽（→10.3）。

</QA>
