<h2><span className="num">12.1</span>权重加载与 tensor-name 约定</h2>

到 Phase 3，我们已经有了单个算子的 CPU/GPU 实现。现在要做的是把它们连接成完整的 Qwen2 网络。 第一个问题是：如何从 safetensors 中找到每个权重？

HuggingFace Qwen2 的 tensor 名字有固定模式。第 $i$ 层的 attention 和 MLP 权重通常如下：

```text
model.layers.{i}.input_layernorm.weight
model.layers.{i}.self_attn.q_proj.weight
model.layers.{i}.self_attn.q_proj.bias
model.layers.{i}.self_attn.k_proj.weight
model.layers.{i}.self_attn.k_proj.bias
model.layers.{i}.self_attn.v_proj.weight
model.layers.{i}.self_attn.v_proj.bias
model.layers.{i}.self_attn.o_proj.weight
model.layers.{i}.post_attention_layernorm.weight
model.layers.{i}.mlp.gate_proj.weight
model.layers.{i}.mlp.up_proj.weight
model.layers.{i}.mlp.down_proj.weight
```

另外还有全局权重：

```text
model.embed_tokens.weight
model.norm.weight
lm_head.weight        # 若 tie_word_embeddings=false 才单独存在
```

因此 `xinfer-model::names` 的作用是把“第几层、哪个权重”转成字符串，避免在 forward 里到处写硬编码路径。

<h2><span className="num">12.2</span>CPU reference forward：整个网络的 oracle</h2>

Phase 2 已经为每个算子写了 CPU reference。Phase 3 要把这些算子串起来，得到完整 Qwen2 forward。 对输入 token 序列 $x_1,\ldots,x_S$，CPU forward 大致是：

$$
X_0 = E[x_1,\ldots,x_S]
$$


  

$$
X_{\ell+1}
      = \operatorname{DecoderLayer}_\ell(X_\ell),
      \qquad \ell=0,\ldots,L-1
$$


  

$$
Z = \operatorname{RMSNorm}(X_L)W_{\text{lm}}^\top
$$

这个 CPU forward 很慢，但非常重要：它是 GPU whole-network forward 的对照组。只测单个 kernel 不够， 因为模型组装可能出错：tensor name 错、转置错、bias 漏加、RoPE 位置错、GQA mapping 错，都可能导致单 kernel 测试通过但整网输出错误。

<h2><span className="num">12.3</span>Resident GPU forward：权重留在 GPU 上</h2>

GPU forward 的第一版采用 resident 模式：所有 layer weights 在模型加载时上传到 GPU，并在整个推理过程中留在显存里。 这样每次 forward 只需要上传输入 token 的 embedding 后 activation，权重不再重复从 CPU 传输。

以一层为例，GPU forward 的顺序与 CPU forward 一致：

1. RMSNorm $X_\ell\rightarrow A$
2. Q/K/V linear + bias
3. RoPE(Q/K)
4. GQA attention
5. o_proj + residual add 得到 $U_\ell$
6. RMSNorm $U_\ell\rightarrow B$
7. gate/up linear、SwiGLU、down linear
8. residual add 得到 $X_{\ell+1}$

<Callout kind="note" tag="为什么 resident 是第一版？">

Resident 模式最简单，最适合验证 correctness。等整网 forward 已经稳定，再引入 layer streaming 等复杂内存策略。

</Callout>

<h2><span className="num">12.4</span>调试 GPU hang：debug layer 与 device removed</h2>

GPU 程序出错时，不一定表现为普通 panic。可能出现 command queue submit 失败、readback 卡住、甚至 device removed。 D3D12 提供两个关键工具：

- **D3D12 debug layer：**输出资源状态、binding、非法 API 使用等诊断信息。
- **`GetDeviceRemovedReason`：**当 device removed 时给出 HRESULT，例如 `0x887A0005`。

在 xinfer 中，`DmlDevice` 增加了 debug message drain 和 removed reason 查询，帮助定位 GPU hang。

<h2><span className="num">12.5</span>解决 DirectML GEMM fault：改用自写 HLSL matmul</h2>

在本项目中，我们隔离出一个问题：DirectML GEMM 在某些 shape 上会触发 device removed，例如 $M=5,K=64,N=32$。 这不是数学公式错，而是特定硬件/驱动/DirectML fallback 路径的问题。

解决方式是：保留 DirectML bring-up 作为学习案例，但模型主路径改用自写 HLSL matmul。后来又进一步优化为：

- f16 weights，降低带宽和显存；
- coalesced GEMV，改善权重读取模式；
- 2D dispatch grid，支持 LM head 的 151936 输出；
- GPU argmax，避免每 token 读回完整 logits。

<h2><span className="num">Lab 12</span>synthetic checkpoint：GPU logits 匹配 CPU reference</h2>

为了避免一开始就调试真实 0.5B 模型，Lab 12 使用一个 synthetic tiny Qwen2 checkpoint： hidden 小、层数少、词表小，但结构完整（GQA、QKV bias、SwiGLU、untied/tied head 都可以覆盖）。

验收目标：

$$
\max_i |z^{gpu}_i-z^{cpu}_i| < \varepsilon
$$

```bash
# 关键验收命令
cargo test -p xinfer-model --test forward_parity -- --nocapture

# 期望看到类似：
gpu_forward_matches_cpu ... ok
kv_cache_decode_matches_full ... ok
streaming_matches_resident ... ok
```

<Callout kind="warn" tag="调试顺序">

如果整网 logits 不匹配，请按 layer 二分：先比较 embedding 后的 $X_0$，再逐层比较 attention output、MLP output、residual。 不要直接从最终 logits 猜问题。

</Callout>

### 思考与练习

<QA question={<><span className="diff easy">基础</span>为什么要用 synthetic checkpoint 做第一轮整网测试？</>}>

synthetic（小而随机初始化的）checkpoint 让你能用极小的层数/维度快速跑通整网前向，并和 CPU 参考逐元素对比，定位组装错误（权重命名、层连接、残差、归一化顺序）。它不依赖下载几百 MB 的真实权重，跑得快、可放进 CI，且因为维度小便于手动核对。等整网 parity 通过后，再换真实 Qwen2.5-0.5B 验证端到端。

</QA>

<QA question={<><span className="diff easy">基础</span>列出 Qwen2 第 0 层 attention 的 8 个主要权重/偏置名字。</>}>

`model.layers.0.self_attn.q_proj.weight` 与 `.q_proj.bias`、`.k_proj.weight` 与 `.k_proj.bias`、`.v_proj.weight` 与 `.v_proj.bias`、`.o_proj.weight`（o_proj 无 bias）。共 7 个 attention 张量；Qwen2 的特点是 q/k/v 都带 bias 而 o_proj 不带。若把同层的 `input_layernorm.weight` 也算上则凑足 8 个。

</QA>

<QA question={<><span className="diff mid">进阶</span>如果忘记加 <code>q_proj.bias</code>，单个 matmul kernel parity 会失败吗？整网 parity 会怎样？</>}>

单个 matmul kernel 的 parity 测试只测 $Y=XW^\top$ 本身，不涉及 bias，所以它仍会通过——bias 是在 linear 之后单独加的。但整网 parity 会失败：少了 q 偏置，Q 数值偏移，经 attention 传播后 logits 与 CPU 参考（含 bias）不一致，max abs diff 超容差。这说明算子级测试通过不代表组装正确，仍需整网 parity 兜底。

</QA>

<QA question={<><span className="diff mid">进阶</span>解释 resident GPU forward 与 streaming forward 的正确性关系。</>}>

两者计算的是同一个数学函数，只是权重的“住处”不同：resident 把所有层权重常驻显存；streaming 把权重留在主机 RAM，按需上传到一个轮转的 GPU slot 池，并用 copy 队列与 compute 重叠。只要上传的内容、绑定和计算顺序正确，二者输出必须逐位（或在容差内）一致。xinfer 用 `streaming_matches_resident` 测试断言 diff=0，即把 streaming 的正确性锚定到已验证的 resident 路径上。

</QA>

<QA question={<><span className="diff hard">挑战</span>设计一个逐层 dump 工具：输出 CPU/GPU 每层 max abs diff，并定位第一层不匹配。</>}>

思路：让 GPU forward 与 CPU 参考 forward 都在每层结束后暴露该层 hidden state。工具按层循环：① 取 CPU 该层输出 $h^{cpu}_\ell$ 与 GPU 输出 $h^{gpu}_\ell$（readback）；② 计算 `max_abs_diff = max(abs(h_cpu - h_gpu))` 并打印 $\ell$ 与该值；③ 设一个容差（如 1e-2，f16 路径放宽），第一处超过容差的层即“首个发散层”，立即报告并可 dump 该层输入与各子模块（norm/attn/mlp）中间量进一步缩小范围。

关键点：逐层比较能把“最终 logits 不对”定位到具体某层某算子；从第一处发散层入手，因为后续层的 diff 是被前面污染的、无独立意义。实现上给 `QwenModel` 加一个可选的 per-layer hook/回调即可。

</QA>
