学习目标

本章把原理图从“已经画完”推进到“允许同步 PCB”。你将运行电气规则检查（Electrical Rules Check，ERC），同时执行 ERC 无法代替的人工审核，并用元件、引脚、网络和警告记录建立发布门。

  • 说明 ERC 能检测什么、不能证明什么；
  • 审核电源方向、输出冲突、未连接引脚、USB-C 与 MCU 引脚映射；
  • 核对数值、封装和符号—焊盘编号；
  • 解释而不是隐藏每一条警告；
  • 用 48/48 元件、172/172 引脚—焊盘和 47 个网络等一致性指标决定是否放行。

17.1 ERC 能发现什么，不能发现什么

ERC 根据符号引脚的电气类型、网络连接和规则配置寻找矛盾。它通常能发现未连接输入、多个主动输出相冲突、电源引脚没有驱动源、网络标志不一致等问题。但它只理解原理图模型；模型错误或规则未覆盖时，ERC 也可能给出“0 错误”。

PCB DRCPCB 数据满足已配置的连通、间距、差分对、尺寸等规则器件本体不碰撞、连接器方向正确、热/信号完整性或实物功能正确
机械审核数据手册本体、庭院、板边、插拔与工具空间满足所用判据网络导通、铜间距、焊接质量或电气性能正确
Gerber/钻孔审核发给板厂的层、板框、孔和实际生成铜符合发布意图原理图逻辑正确、元件会被正确采购或装配板一定工作

任何一列出现“通过”都不能替代其他列。R1 的制造发布已通过设计文件审核，但当前事实库仍有 0 条装配板实测结果。

17.2 电源引脚、输出冲突与未连接引脚检查

先运行严格 ERC，再按类别处理结果。不要从错误总数直接跳到“忽略全部”。

  1. 电源引脚：确认 U1、U2、U3、U4、U5 的电源脚连接到正确轨，电源来源与负载方向合理。
  2. 输出冲突：检查两个推挽输出是否被接到同一网络；UART_TX 与 UART_RX 不应互换或短接。
  3. 双向引脚：根据实际功能审核 MCU GPIO，而不是把所有双向引脚无条件视为安全。
  4. 未连接：九个有意未连接引脚必须显式说明；未标记的悬空脚按错误处理。
  5. 电源标志：确认标志只表达驱动能力，不会把不同电压轨合并。

如果 ERC 报告某电源脚“无驱动”，先确认该网络的真实来源，再决定是否添加正确的电源标志。不要为了清零而把所有电源网络声明成任意驱动源。

17.3 人工复核 USB-C 与 MCU 引脚

USB-C 与 MCU 复用引脚需要逐项对照数据手册和冻结分配表，因为 ERC 不知道“PA11 应当是 D−”这样的产品意图。

  • J1 A6/B6 都进入 D+ 路径，A7/B7 都进入 D− 路径；
  • CC1、CC2 各有独立 5.1 kΩ 下拉，二者不短接；
  • U2 的 I/O 方向与引脚编号正确，保护路径不被旁路；
  • U1 PA11/封装 21 脚接 USB_DM_MCU，PA12/封装 22 脚接 USB_DP_MCU；
  • PA6、PA7、PB0、PB1 分别承担 TIM3 CH1–CH4；
  • PA2/PA3 为 USART2 TX/RX，PA13/PA14 为 SWDIO/SWCLK；
  • PB3/PB4 控制 EN_5V/EN_3V3，PB5 驱动状态 LED。

审核方法：一人读“封装脚号—MCU 引脚—网络—功能”，另一人在原理图和数据手册上逐项指认。不要只检查网络标签文字是否相似。

17.4 复位状态与电源时序审核

原理图审核必须覆盖“固件还没有运行”的时间。目标状态是 GPIO 高阻、PWM 停止、UART_TX 未配置前高阻、5V_SW/3V3_SW 在复位、启动和 USB 挂起期间关闭。

  1. 沿 EN_5V 和 EN_3V3 检查外部下拉，确认 MCU 高阻时仍为低；
  2. 检查 BOOT0 默认下拉与 NRST 上拉/电容，不让上电噪声进入错误启动模式；
  3. 检查目标设备未上电时，GPIO/UART 或受控电源是否可能形成反向供电路径；
  4. 确认 5 V 只作为受控电源输出，不直接连接 3.3 V MCU I/O；
  5. 把“USB 挂起关闭输出”标记为固件和实物待验证要求，而不是原理图已证明事实。

数值审核要同时问“公式是否正确”“工作点是否适用”“布局条件是否满足”。R1 的关键检查包括：

17.6 审核符号到封装的引脚映射

符号正确而封装焊盘编号错误，会产生“原理图看起来完美、PCB 实际接错”的严重故障。审核应从器件实例一路追到封装焊盘。

  1. 确认符号 Device 与 Footprint 属性指向预期的已验证对象；
  2. 比较每个符号引脚编号与封装焊盘编号；
  3. 对 J1、U1、U2、U3、U4、U5、Y1、D1 和有极性电容重点审核方向；
  4. 确认 PTH、NPTH、外壳焊盘和无电气机械脚的类型正确；
  5. 用 PCB 同步后的焊盘网络表反查原理图网络。

最终发布把 48/48 个符号元件映射到 PCB 元件，并把 172/172 个原理图引脚按元件身份、引脚号和网络映射到焊盘。这是强有力的一致性证据，但仍不能证明公共库的机械模型天然正确，因此第 19 章还要审核实体和庭院。

17.7 解释警告，而不是盲目压制

警告可能表示真实风险，也可能是规则无法理解的有意结构。处理原则是：每条警告都要有对象、原因、处置和批准者；“因为太多所以隐藏”不是解释。

  1. 按类别归组，但保留每个受影响对象；
  2. 修复因引脚类型、悬空、命名或电源方向造成的真实问题；
  3. 对有意未连接、DNP 或工具模型限制写明理由；
  4. 只在理由稳定且不会掩盖同类真错误时创建规则例外；
  5. 修改后重新运行严格 ERC，并把结果记录在发布清单。

R1 最终严格原理图 ERC 为 0 个错误、2 条非阻断提示；当前 API 没有暴露提示正文，因此发布资料如实记录其数量和“不可读取正文”的限制，而不是伪造解释。若在交互界面可读取，应逐条补录。

同步前先建立基线，之后每次 PCB 更新都与基线比较。单看“同步成功”不能排除漏件、错网或 DNP 属性变化。

17.9 PCB 同步前的质量门

只有以下项目全部满足，原理图才可同步到 PCB：

  • 严格 ERC 没有未解决错误；每条警告/提示都有记录；
  • USB-C、MCU 复用引脚、晶振、复位和调试映射已人工双人复核；
  • VBUS_IN、3V3_SYS、3V3_SW、5V_SW、GND 保持独立；
  • 最终器件值和封装已与数据手册及工程源一致；
  • J2/J3 的 DNP 属性明确，九个未连接引脚已说明；
  • 48 个元件、172 个引脚—焊盘和 47 个网络的基线已保存；
  • 保存、关闭并重新打开原理图后，复查结果不变；
  • 审核者签名、日期、工程归档和 ERC 报告已关联到同一修订。

复习题：为什么 ERC 为 0 仍可能存在电源轨塌缩？为什么 172/172 映射仍不能证明连接器能插入？哪一种证据能确认板厂看到的实际地铜？

本章小结

ERC 是原理图质量门的一部分，不是万能认证。可靠放行还需要人工引脚与状态审核、数值和封装复核、警告解释，以及原理图—PCB 数量与网络基线。通过本门只说明设计可以进入物理实现，尚未证明布局、制造或实物性能。

事实依据：R1 制造检查清单、最终工程归档和项目事实库。ERC/映射结果为设计发布验证；硬件电气、热、USB 和固件行为仍待实测。
